2. What This Flow Controls

The current reload workflow contains hardcoded API calls against two n8n workflow IDs:

Controlled workflow IDObserved action in reload flowCross-file identification

Important distinction: the reload flow is not the WebSocket Handler and not the Downloader. It sits one level above them and manages their process lifecycle through the n8n API.

3. Entry Trigger and Overall Shape

The entire workflow starts from a single Schedule Trigger configured with an hourly interval. From that trigger, execution immediately splits into two parallel branches.

Schedule Trigger (hourly)
  |- Branch A: restart workflow 116 (WebSocket Handler)
  |  -> deactivate 116
  |  -> wait 10 sec
  |  -> activate 116
  |  -> if activation != 200, retry activation loop
  |
  |- Branch B: inspect workflow 118 (Downloader)
     -> query latest execution of 118
     -> if latest stoppedAt is older than now - 2 hours
        -> deactivate 118
        -> wait 10 sec
        -> activate 118

4. Branch A — Forced Refresh of Workflow 116

The first branch is unconditional. Every time the hourly schedule fires, it posts to the n8n API to deactivate workflow 116, waits 10 seconds, then posts again to activate workflow 116.

Proven behavior: workflow 116 is deliberately bounced on every hourly run of the reload flow. There is no age check and no dependence on execution history for this branch.

4.1 Retry logic on the 116 restart path

The activation response for workflow 116 is explicitly examined. If the response code is not 200, the true output of IF Error goes back to the same Wait node, creating a retry loop that repeatedly attempts activation after 10-second pauses.

Deactivate 116
  -> Wait 10 sec
  -> Activate 116
  -> IF statusCode != 200
       true  -> Wait again -> Activate again -> ...
       false -> Set Done = true
Important asymmetry: the workflow explicitly retries activation of 116, but it does not check the deactivation response first. So the supervision logic is primarily concerned with ensuring that 116 ends in an activated state.

5. Branch B — Conditional Recovery of Workflow 118

The second branch is conditional. It first queries the n8n executions API for the most recent execution of workflow 118, limited to one result. Then it checks the latest execution’s stoppedAt timestamp against $now.minus({ hours: 2 }).

Proven behavior: workflow 118 is not always restarted. It is restarted only when the reload flow concludes that the latest recorded execution for 118 is older than two hours.

5.1 Naming mismatch in the IF node

The node is named If Last Execution Hour Ago, but the expression inside it compares against $now.minus({ hours: 2 }).

Confirmed mismatch

The node name suggests a one-hour threshold, but the actual condition uses two hours. The JSON therefore proves that the behavior follows the expression, not the label.

5.2 No visible success/error handling after Activate1

Unlike the 116 path, the 118 path does not route HTTP Request Activate1 into any downstream IF or Set node. The activation request is made, but the current reload workflow export does not show an explicit branch that validates the response or retries if activation fails.

Second important asymmetry: workflow 118 has a conditional restart and a fire-and-forget activation, while workflow 116 has an unconditional restart and an explicit activation retry loop.

6. Why Workflows 116 and 118 Matter Here

Cross-file project notes identify workflow 116 as the Fortnox WebSocket Handler and workflow 118 as the Fortnox Downloader.

WorkflowRole in broader Fortnox architectureWhy a reload flow may supervise it
116 — WebSocket Handler Real-time event listener and router for Fortnox WebSocket topics Long-lived socket listeners often benefit from periodic refresh or reconnect
118 — Downloader Batch paginator/state machine for Fortnox list backfills and reconciliation sweeps A stalled or stale downloader can leave the Settings-driven batch cycle parked mid-process

Likely operational intent

The most plausible design intent is:

  1. keep the WebSocket listener fresh by forcibly reconnecting it every hour, and
  2. heal a stale downloader only when its latest recorded execution appears too old.

This intent is not spelled out in the JSON, but it fits the exact branch logic and the known roles of workflows 116 and 118.

7. What Is Proven vs. What Is Inferred

7.1 Proven from the current reload JSON

  • The reload flow is triggered on an hourly schedule.
  • It controls workflows 116 and 118 through the n8n REST API.
  • Workflow 116 is always deactivated, then reactivated after a 10-second wait.
  • Workflow 116 activation is retried until the activation HTTP status code becomes 200.
  • Workflow 118 is only restarted when the latest fetched execution has stoppedAt older than two hours.
  • The node name If Last Execution Hour Ago does not match its actual two-hour threshold.
  • The 118 restart path has no explicit downstream retry/validation node after activation.

7.2 Confirmed from prior uploaded workflow analyses

  • Workflow 116 is the Adgrabber Fortnox WebSocket Handler.
  • Workflow 118 is the Adgrabber Fortnox Downloader.
  • The WebSocket Handler is a real-time Fortnox event router.
  • The Downloader is a batch orchestrator/state machine driven by an Airtable Settings record.

7.3 Inferences that remain plausible but not fully proven

Inference A — this flow is a resilience / self-healing layer

The reload flow looks like an ops safeguard added on top of the main Fortnox sync architecture. That interpretation is strong, but the JSON export itself does not contain a human note or comment confirming that intent.

Inference B — workflow 116 is treated as more fragile or more important to keep hot

The stronger restart discipline for 116 suggests that the WebSocket listener is treated as a process that should always be actively refreshed, while 118 is treated as a background worker that only needs intervention when stale. This is a reasonable reading of the asymmetry, but still an inference.

Inference C — the age check for 118 is a coarse health signal rather than a true stall detector

The logic does not inspect execution status, error state, queue state, or runtime duration. It only looks at the latest stoppedAt timestamp. So the health check is simple and coarse. That conclusion follows from structure, but the intended SLA is not documented in the export.

10. Plain-English Explanation

If you explain this flow to a teammate without node jargon, the clearest summary is:

Every hour, this watchdog flow does two things:
1. It force-restarts the WebSocket listener workflow so the live Fortnox connection is refreshed.
2. It checks whether the batch downloader workflow looks stale.
   If the downloader's latest recorded execution is older than two hours,
   it restarts that workflow too.
Best one-sentence description: this workflow is an ops watchdog that keeps the Fortnox sync machinery alive by periodically refreshing the live listener and conditionally restarting the batch backfill worker.

11. Current State of Understanding

  • The reload workflow is active and self-contained.
  • It supervises workflow 116 and workflow 118 through hardcoded n8n API endpoints.
  • Workflow 116 is restarted on every hourly trigger and has explicit activation retry handling.
  • Workflow 118 is only restarted when the most recent execution appears older than two hours.
  • The reload flow therefore complements the main sync architecture rather than replacing it.
