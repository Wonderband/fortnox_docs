Key Conclusion

Fortnox supplier invoices can and do carry client/campaign information — via Fortnox Project codes assigned by Adgrabber's bookkeeper when processing invoices inside Fortnox. This allocation mechanism was working in the past and produced per-campaign row breakdowns on supplier invoices. It appears to have stopped being used for recent invoices. It can be reactivated, and combined with an n8n automation, could eliminate most manual work in sys.fakturamapping.

1 — What are Fortnox Projects?

Fortnox Projects (Pr) are an accounting dimension built into Fortnox. Each project has a project number (free-form, up to 20 characters) and a name. You create them once in the Fortnox register, then tag individual accounting rows with that code to track where money is going.

Projects can be assigned to three document types in Fortnox:

Document type How you assign a project Effect
Customer invoice / order PR column on each line item, or at document level Revenue is tracked against the project
Supplier invoice PR column on each accounting row, or globally under "Ytterligare information" Cost is tracked against the project
Journal entry (verifikat) PR column on each row Any debit/credit tracked per project
Important for Adgrabber: When a bookkeeper processes a supplier invoice in Fortnox, they can add a project code to each row — one row per campaign. This is exactly what was happening with older invoices (project codes 137–520 range). The project name in Fortnox would say something like "Fritidscenter kampanj Q1" or the client name, giving a human-readable meaning to the number.

2 — Two sources of records in Airtable

Source A — Fortnox API sync (n8n)
  • @url = api.fortnox.se/3/supplierinvoices/…
  • Created By = integration adgrabber
  • SupplierInvoiceRows JSON blob present
  • Invoice created by supplier in their system, imported into Fortnox
  • Bookkeeper in Fortnox assigns project codes per row → synced to Airtable
Source B — Manual / Platform Import Files
  • No @url, no SupplierInvoiceRows blob
  • Created By = human user name
  • Created directly in Airtable or imported from Platform Import Files table
  • No project codes available — allocation always 100% manual

The project-code mechanism described in this document applies only to Source A records. Source B records always require full manual allocation.

4 — How the system works: a real example

Below is a simplified reconstruction of how a Hemnet invoice would have been processed when the project-code workflow was active. The data comes from actual Airtable records.

Step 1 — Supplier invoice arrives in Fortnox
Hemnet AB
Invoice for display advertising — January
Fortnox GivenNumber: XXX
Total: 12,363 SEK incl. VAT
Account Description Code Project (PR) Amount
2440 Leverantörsskulder TOT 471 −12,363

4611 Direkta kampanjkostnader i Sverige, 25% moms PRE 471 +9,890.40

↑ Bookkeeper manually enters "471" in the PR column for each row, meaning: this spend belongs to project 471.

Step 2 — n8n syncs the invoice to Airtable

The SupplierInvoiceRows field in Airtable stores the raw JSON from Fortnox:

[
  { "Account": 2440, "Code": "TOT", "Project": "471", "Total": -12363 },
  { "Account": 2641, "Code": "VAT", "Project": "471", "Total": 2472.6 },
  { "Account": 4611, "Code": "PRE", "Project": "471", "Total": 9890.4 }
]

The project code "471" is preserved in the JSON. It's already in Airtable — no extra API call needed to read it.

Step 3 — Parse project code → look up project name

n8n reads the JSON, extracts the unique project codes, and calls the Fortnox Projects API:

GET https://api.fortnox.se/3/projects/471

Response:
{
  "Project": {
    "ProjectNumber": "471",
    "Description": "Fritidscenter i Göteborg AB — Kampanj Q1 2020",
    "Status": "ONGOING"
  }
}
Step 4 — Match to Airtable → auto-create sys.fakturamapping

The project description is matched (by name or a stored code) to a record in Media / Prod New. A sys.fakturamapping entry is created automatically:

471 → Fritidscenter — Q1 2020 → Media/Prod New: recXXXXXX 9,890 SEK

The 9,890 SEK (account 4611, the actual campaign cost row) is allocated. VAT and payable rows are ignored — only cost rows (account 4xxx) are relevant for spend tracking.

Bidtheatre is a programmatic advertising platform. They send Adgrabber one consolidated invoice per month covering all campaigns they ran for all Adgrabber clients. In the Fortnox accounting, the bookkeeper breaks this down: one accounting row per client project, each tagged with a different project code.

5 — Current status of the system

What was working (older invoices)
  • Project codes 137–520 range found in data
  • Multiple suppliers using it: Hemnet, Bidtheatre, Reklambossen, Westcoast Digital
  • Per-row breakdown already done in Fortnox by bookkeeper
  • Data is in Airtable — in the SupplierInvoiceRows JSON blob
  • It was never auto-synced to sys.fakturamapping — always manual
What is not working (recent invoices)
  • Recent supplier invoices have Project: "" on all rows
  • Recent FortnoxOrders (4000+) also have empty Project fields
  • The bookkeeper stopped tagging project codes at some point
  • Exact date of change unknown — no meta-information available
  • sys.fakturamapping entries still being created, but 100% manually
The project codes were never the bottleneck. Even when bookkeepers were tagging invoices with project codes in Fortnox, that data was never read by n8n and never used to auto-populate sys.fakturamapping. The automation potential existed but was never built.

7 — Can the system be reactivated? What needs to happen?

Yes — the system can be reactivated. It requires two parallel tracks: one human workflow change (bookkeeper) and one n8n automation.

1

Understand what project codes mean

Call GET /3/projects on the Fortnox API to get the full list of Fortnox Project names. This tells us whether codes 303, 394, 462 etc. are named after clients, campaigns, or something else. This is the prerequisite for building a mapping to Airtable records.

2

Resume bookkeeper tagging in Fortnox (human step)

When processing a supplier invoice in Fortnox, the bookkeeper must fill the PR column on each accounting row with the relevant Fortnox Project code. This is the only human input required — one project code per row, entered once in Fortnox. No changes to Airtable workflow needed from their side.

3

Store a project → Airtable record mapping

Create a lookup table (could live in Airtable or in n8n) that maps each Fortnox Project number to the corresponding Media / Prod New record ID. This mapping only needs to be built once and updated when new projects are created.

4

Build n8n automation: parse → match → create

When n8n syncs a supplier invoice to Airtable, it additionally: (a) parses the SupplierInvoiceRows JSON, (b) groups rows by Project code, (c) sums the amount per project (account 4xxx rows only), (d) looks up the Airtable record for each project, (e) creates sys.fakturamapping entries. Invoices where any Project code is missing or unrecognised are flagged for manual review.

5

Handle the "no project code" case

For invoices where the bookkeeper did not tag project codes (or suppliers who send undivided invoices), the existing manual sys.fakturamapping workflow remains as a fallback. The Supplier Usually Needs Allocation flag already identifies which suppliers typically need this treatment.

Requirement Who Effort
Fetch Fortnox project list to understand naming Developer / analyst One API call
Bookkeeper resumes PR column tagging in Fortnox Accounting team Process change — ongoing
Build project → Airtable record mapping table Developer + accounting team One-time setup, ~1–2 days
n8n workflow: parse JSON → create fakturamapping Developer New n8n flow, ~2–3 days
Backfill old invoices that already have project codes n8n automation Can be done immediately for existing data
