Supplier Invoice Allocation — Research & Proposed Automation

Adgrabber · Base appk463NGeDy17j9T · FortnoxSupplierInvoice tblYERBn2P6wrHEzl · sys.fakturamapping tblRXZqzj79Vd3iC6 · Media/Prod New tblTf9sNLy464khq5 · 2026-03-26

Summary

The goal is to automatically create sys.fakturamapping records linking supplier invoices to specific client campaign line items (Media/Prod New) — currently done 100% manually. Research shows a previous partial attempt using Fortnox Project codes was architecturally insufficient: project codes identified clients/orders but not specific campaigns, and the bookkeeper stopped tagging them. A revised strategy — where each Fortnox Project maps directly to one Media/Prod New record — removes all ambiguity and enables full automation with a one-time setup effort.

1 — The Problem: Manual sys.fakturamapping

When Adgrabber receives a supplier invoice (e.g. from Google, Bidtheatre, Hemnet), the cost must be allocated to the specific client campaigns the spend belongs to. This allocation is stored in TABLE sys.fakturamapping: one record per invoice-per-campaign-line, linking the FortnoxSupplierInvoice to the Media/Prod New record (campaign line item) and recording the amount.

A single consolidated supplier invoice may cover costs for many clients and many campaigns at once. Today, a staff member manually creates one sys.fakturamapping record per allocation split. For a Bidtheatre invoice covering 24 campaigns, that is 24 manual records.

The target is to replace the manual step with an n8n automation that reads the supplier invoice and automatically creates the correct sys.fakturamapping records with no human input.

3 — What is in a Supplier Invoice Row? (SupplierInvoiceRows JSON)

Every Source A invoice stores a SupplierInvoiceRows JSON array in Airtable, mirroring the Fortnox accounting rows. A typical invoice for one client looks like this:

4 — What Fortnox Projects Are

Fortnox Projects (PR) are an accounting dimension. Each project has a number and a name, created in the Fortnox register. The bookkeeper can tag any accounting row on any document with a project code. 326 projects exist in Adgrabber's Fortnox account (verified by inspection of the project register).

Fortnox projects are assigned via the PR column on supplier invoice rows. They track where costs go within the company's accounts.

Key finding from Fortnox project register inspection: Project names follow no consistent convention. Examples found in the register: Order 462 - Fritidscenter i Göteborg AB, Omnicom Media Group - Order 418, Fritidscenter-495, Juni Sociala medier Vätterhem, Mars Växjö, 523. Naming was entered manually with no enforced format. The project name alone is not a reliable machine-readable key.

Customer link: some projects have a Fortnox customer formally linked; others have an empty customer field (e.g. project 495 = "Fritidscenter-495" has no customer linked, only the name suggests the client).

5 — How the Old Project Code Approach Worked (and Why It Is Insufficient)

Between approximately 2019 and 2021, the bookkeeper tagged invoice rows with project codes. These codes were found in the SupplierInvoiceRows JSON blob on older invoices (project numbers 137–520 range). The data was never used by n8n — sys.fakturamapping remained 100% manual. After 2021, the bookkeeper stopped tagging entirely.

Why the old approach cannot be made to work reliably

Fortnox project codes in the old system identified a client or order, not a specific campaign. Matching from project code to Media/Prod New record requires: (1) the supplier name to derive channel type (not derivable from supplier name alone — needs a human-maintained lookup table per supplier), and (2) the channel type + client + date to be unique within Media/Prod New. Case 2 above shows that condition (2) regularly fails for clients with multiple concurrent campaigns of the same type. Even if both conditions were met, they represent two fragile inferred steps, not a direct key.

7 — The Proposed Strategy: Fortnox Project = Media/Prod New Campaign

Core Principle

Change what a Fortnox Project represents: from "this client/order" to "this specific Media/Prod New record". Each active campaign in Airtable gets its own dedicated Fortnox project with the M-number as the project code. The bookkeeper tags invoice rows with the M-number. n8n reads M-number → finds Airtable record directly → creates sys.fakturamapping. No inference, no lookup tables, no ambiguity.

8 — What Needs to Change

1

Add FortnoxProjectCode field to Media/Prod New in Airtable

A text field storing the Fortnox project number for each campaign record (e.g. M4329). This is the link between Airtable and Fortnox. Populated automatically when the project is created (step 2).

2

n8n: auto-create Fortnox project when a campaign is activated

When a Media/Prod New record becomes active, n8n calls POST /3/projects with:

  • Number — the M-number (e.g. M4329, or numeric 4329 if Fortnox rejects the prefix — confirm with one test call)
  • Description — constructed name: M4329 - [ClientName] - [CampaignType] - [Month]
  • ContactId — Fortnox customer number as a plain string (e.g. "10042"), pulled from the linked FortnoxCustomer record. Not an object — Fortnox REST API takes plain ID strings for all relations.
  • ContactType — "CUSTOMER"

On success, the API returns the created project. n8n writes the project number back to the FortnoxProjectCode field in Airtable. This write-back is a confirmation flag: empty = project not yet in Fortnox (n8n will retry); populated = project exists (n8n skips). It also shows the bookkeeper which code to use in the PR column.

Trigger mechanism — two options

Option How it works Consistency with current system
A — n8n polls Airtable Recommended n8n Schedule trigger (e.g. every 15 min) queries Media/Prod New for records where Status = Active AND FortnoxProjectCode is empty. Processes each match, writes project number back. Identical to the existing Airtable → Fortnox push pattern already designed in the system: Sync Fortnox and Order Status fields on FortnoxOrder use the same write-queue flag approach — n8n polls, acts, marks as done. No new infrastructure.
B — Airtable automation pushes to n8n webhook Airtable native automation fires when Status changes to Active → calls an n8n Webhook node URL. n8n processes the payload immediately. Not used anywhere in the current system. Introduces a new pattern (Airtable as the caller, n8n as the listener). Near-real-time but adds Airtable automation maintenance overhead.

Note: Fortnox does now support webhooks (section 8.4 of the integration docs), but only for integration lifecycle events (purchase, licence expiry, user invitations) — not for order or invoice events. Fortnox webhooks are not applicable here.

3

Bookkeeper process: use M-number project codes when processing invoices

When processing a supplier invoice in Fortnox, the bookkeeper fills the PR column on each accounting row with the M-number of the campaign. The Fortnox project list (auto-populated by step 2) shows readable names: "M4329 - Fritidscenter - Facebook/Instagram - 2022-05" — the bookkeeper selects from this list, no guessing required. This is the only human input in the whole process.

4

n8n: parse SupplierInvoiceRows → create sys.fakturamapping

When n8n syncs a supplier invoice to Airtable, an additional automation runs: (a) parse SupplierInvoiceRows JSON, (b) filter to account 4xxx rows, (c) group by Project code, (d) sum Debit per project, (e) for each project code matching M\d+, look up the Media/Prod New record where Autonumber = that number, (f) create one sys.fakturamapping record: link to FortnoxSupplierInvoice + link to Media/Prod New + Belopp ex moms = summed amount. Rows with no project code, or unrecognised codes, are flagged in a review queue.

5

Fallback: invoices with no project codes remain manual

Any invoice where the bookkeeper did not fill the PR column (or filled it with a non-M code) is flagged with a status in Airtable indicating it needs manual allocation. The existing manual sys.fakturamapping workflow is the fallback. The field Supplier Usually Needs Allocation on FortnoxSupplier already identifies which suppliers require allocation treatment.

9 — Limitations and Edge Cases

SituationHow handled
Invoice has no project codes (bookkeeper skipped PR column) Flagged for manual allocation. Existing manual workflow applies.
Invoice row has a non-M project code (old-style client codes like "462") Code not recognised as M-number → flagged for manual review.
One invoice covers campaigns that span multiple months Staff creates multiple rows in Fortnox with the same M-number, or a single row tagged to the correct M-record. No issue — the M-number is the only key needed.
Platform import invoices (Facebook, LinkedIn, etc.) Out of scope — these are Source B invoices, always allocated manually per existing workflow.
Campaign is cancelled before invoice arrives Fortnox project still exists; n8n finds the Airtable record (now closed). Staff may need to handle status manually.
A single invoice row covers multiple campaigns (no per-campaign breakdown) Bookkeeper must add one row per campaign in Fortnox when processing. This is already the expectation for the PR column workflow.
