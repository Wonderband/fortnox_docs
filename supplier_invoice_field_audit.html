This table stores supplier invoices from two distinct sources: (1) invoices synced automatically from Fortnox via n8n, and (2) platform invoices (Facebook, Google, TikTok, etc.) entered manually by staff via the platform import workflow. These two record types share one table but have structurally different field populations — many fields only apply to one source type.

A — Fortnox-Side Fields  Origin: Fortnox / n8n sync

These fields all originate from the Fortnox accounting system. n8n polls GET /3/supplierinvoices and upserts records in Airtable using FIELD @url as the upsert key.

Sync gate: n8n only processes invoices where the linked FortnoxSupplier has FIELD Internal Status = "Sync". Suppliers marked "Do not sync" are skipped entirely.

⚠️ Any manual edit to an A-field in Airtable will be silently overwritten on the next sync. Changes must be made in Fortnox first.

Note on text-typed fields: Fortnox returns all values as strings. n8n writes them as-is, so dates and numbers are stored as singleLineText. Formula fields in Category D parse these into usable types.

Caveat on A1/A2 completeness: Classification is based on the Fortnox REST API spec. Not every field in the spec is written by n8n — VoucherNumber, VoucherSeries, VoucherYear exist in the Fortnox API and are marked A1, but n8n does not write them to Airtable. Voucher details are only available inside the Vouchers JSON blob. Other optional A2 fields (YourReference, CostCenter, Project, etc.) may be empty on most records simply because staff did not fill them in Fortnox.

A1 — Read-only in Fortnox API  Fortnox calculates · staff cannot change

These fields are marked read-only in the Fortnox API spec. Their values are calculated or assigned by Fortnox's accounting engine — the document number it assigns, the amounts it calculates, the vouchers it creates when booking, the payment status it tracks. Adgrabber staff cannot change these values even within Fortnox itself. n8n is the sole writer in Airtable, and what it writes is always exactly what Fortnox computed.

A2 — Writable in Fortnox API  Staff entered this value in Fortnox

These fields are writable in the Fortnox API — meaning staff at Adgrabber (or the supplier) provided the value when the invoice was created or updated in Fortnox. n8n copies whatever is stored there into Airtable on each sync. The value reflects a human decision, not a machine calculation.

Practical implication: if the value looks wrong in Airtable, the fix must be made in Fortnox — not in Airtable. Any Airtable edit will be overwritten on the next sync. These fields are also the ones most likely to be empty or inconsistently filled (staff don't always fill every optional field).

Key Structural Observations

0. Not all "Fortnox fields" are equal — the A1/A2 distinction matters

Category A is split into two subcategories. A1 (24 fields) = read-only in the Fortnox API — Fortnox's accounting engine generates the value (amounts, status flags, assigned numbers). Staff cannot change these even in Fortnox. A2 (16 fields) = writable in the Fortnox API — staff typed these values when registering the invoice in Fortnox. Important caveat: VoucherNumber, VoucherSeries, and VoucherYear are A1 fields (Fortnox generates them) but n8n does not write them to individual Airtable fields — voucher details are only available inside the Vouchers JSON blob.

1. Two record types sharing one table

This table conflates two structurally different record types: Fortnox-synced invoices (where @url is set, FortnoxSupplier is linked, all Fortnox fields are populated) and platform import invoices (where @url is empty, FortnoxSupplier is empty, Kund + Platform Import File are populated instead). There is no explicit type discriminator field — record type must be inferred from which fields are populated.

2. Date fields stored as text throughout

InvoiceDate, DueDate, FinalPayDate, Balance, and CurrencyRate are all stored as singleLineText. This is a consequence of how n8n writes Fortnox API responses. Three separate formula fields parse InvoiceDate into a proper date — all three are identical duplicates.

3. 23 fields are candidates for removal

5 are outright broken (return no data). 23 are duplicates, dev artifacts, or dead fields (never populated). Removing all of them would reduce the table from 120 to approximately 92 fields with no functional loss.

4. The mapping integrity chain

The Ek. utfall. Externkostnad rollup is the final step of the cost attribution chain: FortnoxSupplierInvoice → sys.fakturamapping → Media/Prod New. The "Invoice fully mapped?" formula compares this rollup against "Total Excl VAT in SEK" to detect gaps. This is the most important computed value for operational use.
