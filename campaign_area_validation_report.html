Campaign–Area Budget Validation:
Why Forms and Record Detail Layouts Behave Differently

Project: DEV. Betaportal Adgrabber 2.0  ·  Interface: Admin (Dev 4.2)  ·  Date: 2026-03-17

1Context & Setup

The Kampanjer table has a linked record field Area pointing to the Areas table. A conditional dynamic filter is applied to that field in the Interface Designer:

Filter rule on the Area field (Kampanjer table)
Area Remaining Budget ≥ Brief. Ek. Kampanjbudget

Intent: a user must not be allowed to link a Campaign to an Area whose remaining budget is less than the campaign's planned budget.

There are two paths to create or edit a campaign inside the Admin (Dev 4.2) interface:

Path Interface / Layout Validation on submit / save
Path A — Form
Budget & Areas → "Add Campaign"		 Budget & Areas interface
Interface Form layout (Add Record form) 		✅ Full — blocks submit if condition broken
Path B — Record Detail
Kunder → Kunder Detail → Kampanjer row → Campaign Details		 Kunder Detail (correct)
→ Campaign Details (Correct) 		❌ Partial — filters picker, does not block save

2What the Conditional Filter Actually Does

The "Filter record selection by a condition" setting is a picker display filter, not a database constraint. Airtable's own documentation describes it exclusively in terms of what is shown in the selection UI:

"Filters for record selection — Omit any records using static, dynamic, or both conditions."

"Dynamic filtering — Limits the available options based on the current record." Source: support.airtable.com/docs/dynamic-filtering-in-linked-record-fields

Note the language: "limits the available options", "omit records". There is no language such as "prevents saving", "blocks the record", or "enforces at commit time". The filter controls what the user sees in the dropdown; it has no power over what gets written to the database.

Key insight
The conditional filter is a UI display filter, not a data validation rule. Whether it feels like full validation depends entirely on whether the surrounding UI provides an escape path.

3The Core Architectural Difference

3.1 Interface Form — Atomic Submission

An Interface Form (Add Record) is a single transactional unit. Every field the user fills in lives in an in-memory draft until the Submit button is pressed. At that point, Airtable evaluates all field conditions against the current draft values as a whole before writing anything to the database.

1. Fill fields
All values held in memory (not saved yet)
2. Select Area
Picker only shows Areas passing the filter vs. draft budget value
3. Raise budget
Budget draft value increases; filter re-evaluates dynamically
4. Click Submit
Airtable re-checks all conditions — selected Area no longer passes → submission blocked

The official documentation confirms this submit-gate behaviour:

"If any fields error out for form submitters, then they will be unable to submit the entry until the error is resolved." Source: support.airtable.com/docs/building-and-sharing-forms-in-airtable

Additionally, forms explicitly do not allow creating new linked records inline:

"Linked record fields included in an interface form will allow users to select existing linked records, but they won't be able to create new linked records. For that, users would need to create the linked record in another interface form first and then reload the interface page to select the newly created linked record." Source: support.airtable.com/docs/airtable-interface-layout-form

This means both bypass paths are closed in a form: No "create new Area" button Submit blocked if condition broken

3.2 Record Detail Layout — Per-Field Auto-Save

A Record Detail layout edits an already-existing record field by field. There is no Submit button. Every field change is committed to the database immediately and independently the moment the user confirms that field's edit. There is no moment where all fields are evaluated together.

1. Open record
Campaign record already exists in DB
2. Link Area
Picker shows filtered Areas; user selects one → saved immediately
3. Edit budget
Budget field updated → saved immediately — no awareness of Area field
4. Nothing fires
No transaction boundary → the broken condition is never re-evaluated → DB now has inconsistent data

The record detail documentation describes fields as directly editable with no submission step:

"Link / unlink records — When this setting is toggled on, end users with editor permissions can: Link or unlink existing records in the record detail page." Source: support.airtable.com/docs/airtable-interface-layout-record-detail
"Add records through a form — Toggling on this option will allow end users with editor permissions to add a new linked record via an interface form layout." Source: support.airtable.com/docs/airtable-interface-layout-record-detail

Notice: adding a new linked record from a record detail routes through a separate form layout — it does not happen inline — confirming that the record detail itself has no submission/validation lifecycle.

Root cause in Campaign Details (Correct)
The Campaign Details (Correct) layout has an Area linked record element with "Allow users to create new areas" toggled ON. This makes the loophole doubly open:
  1. A user can raise the campaign budget in Brief. Ek. Kampanjbudget → saved instantly, no validation.
  2. A user can create a brand-new Area record with an arbitrary budget directly from the picker → the new Area has no history, the condition is never tested against it.

4Worked Examples

Example A — Form path (Budget & Areas interface) ✅

StepWhat happensResult
1. Open "Add Campaign" Interface Form opens. All values are in-memory draft. —
2. Enter Brief. Ek. Kampanjbudget = 80,000 SEK Draft value stored. Filter Area Remaining Budget ≥ 80,000 activates. Picker shows only Areas with ≥ 80,000 remaining.
3. Select "Kongahälla Center Q2" (100,000 remaining) Area linked in draft. Condition passes. —
4. Increase budget to 120,000 SEK Draft budget rises. Filter now ≥ 120,000. "Kongahälla Center Q2" only has 100,000 remaining. Form marks the Area field as invalid.
5. Click Submit Airtable re-evaluates all conditions on the draft. Area fails. ❌ Submit blocked. "This option is not available to select."
6. User must fix Either reduce budget ≤ 100,000 or select a different Area with ≥ 120,000 remaining. ✅ Data integrity enforced.

Example B — Record Detail path (Campaign Details layout) ❌

StepWhat happensResult
1. "Create new Campaign" button Automation creates a blank Kampanjer record. User navigates to Campaign Details (Correct). —
2. Enter Brief. Ek. Kampanjbudget = 80,000 SEK Field saved to DB immediately. No cross-field check. DB: budget = 80,000. Area = empty.
3. Open Area picker Picker reads budget from DB (80,000). Shows Areas with ≥ 80,000 remaining. Filter is working correctly here. —
4. Select "Kongahälla Center Q2" (100,000 remaining) Area linked and saved immediately to DB. DB: budget = 80,000, Area = Kongahälla (valid).
5. Increase budget to 120,000 SEK Budget field saved immediately to DB. No validation fires. The already-linked Area is not re-checked. ❌ DB: budget = 120,000, Area = Kongahälla (only 100,000 remaining). Inconsistent.
6. Create new Area with 1 SEK (loophole) "Allow users to create new areas" is ON. User creates "Fake Area" with 1 SEK budget from within the picker. Immediately linked. Condition never evaluated against it. ❌ DB: budget = 120,000 linked to Area with 1 SEK remaining. Severely inconsistent.

5Why the Difference Exists — Summary

Aspect Interface Form (Add Record) Record Detail Layout
Save model Atomic: one transaction on Submit Per-field: each edit auto-saves independently
Validation moment At submit — all fields evaluated together None — no submit step exists
Conditional filter scope Evaluated against in-memory draft values at submit Evaluated against saved DB value only when picker is opened
Cross-field re-evaluation ✅ Yes — changing budget updates filter AND blocks submit if broken ❌ No — changing budget has no effect on already-linked Area
Create new linked record inline ✅ Not possible (by design, per Airtable docs) ❌ Possible — "Allow users to create new areas" toggle exists and is currently ON
Net result for validation ✅ Full budget validation enforced ❌ Filter works on picker only; no protection after linking

6Recommendations (if Record Detail is kept)

If you choose to keep the Campaign Details (Correct) record detail layout as the primary campaign editing path, the following measures are recommended in order of effectiveness:

Fix 1 — Disable "Allow users to create new areas" (Immediate, do this now)

In Interface Designer → Admin (Dev 4.2) → Kunder Detail (correct) → click Kampanjer grid → Customize "Campaign Details..." → click the Area element → right panel → Permissions → "Allow users to create new areas" → turn OFF.

This closes the most egregious bypass (creating fake Areas with 1 SEK). The conditional filter on existing records will continue to work correctly.

Fix 2 — Add a formula warning field (Visible safety net)

Create a formula field in Kampanjer, e.g. Budget Overrun Warning: 

IF(
  AND(
    {Area Remaining Budget} < {Brief. Ek. Kampanjbudget},
    {Area} != ""
  ),
  "⚠️ Area remaining budget (" & {Area Remaining Budget} & " SEK) is less than campaign budget (" & {Brief. Ek. Kampanjbudget} & " SEK)",
  ""
)

Add this field to the Campaign Details (Correct) layout. Users will see a visible red warning whenever the budgets are inconsistent, even if saving is not blocked.

Fix 3 — Automation validation (Enforce on any save)

Create an Airtable Automation:

  • Trigger: When Kampanjer record is updated (watch fields: Area, Brief. Ek. Kampanjbudget)
  • Condition: Area Remaining Budget < Brief. Ek. Kampanjbudget AND Area is not empty
  • Action: Send notification to MGMT Ansvarig, or update a status field to "Budget Overrun – Needs Review"

This cannot prevent the save, but it creates an immediate alert so the problem is caught right away rather than discovered later.

Fix 4 — Route Area selection through a form (Maximum enforcement)

Remove the Area element from Campaign Details (Correct) entirely. Force users to set or change the Area only through the validated Budget & Areas Add Campaign form. This is the most restrictive option but guarantees no bypass is possible.

Recommended minimum action
Apply Fix 1 (disable create new Areas) immediately — it requires no schema changes, no automations, and closes the biggest loophole in under 30 seconds. Add Fix 2 (formula warning) as a visible safety net for the remaining gap (editing budget after Area is already linked).
