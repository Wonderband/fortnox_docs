Airtable-only

Recalc
This workflow recalculates accrual-style deltas and writes them back.

The workflow is a focused Airtable calculation job. It loads open Media / Prod New records, supplier-cost mappings from sys.fakturamapping, and invoiced allocation records from sys.orderallocationmapping. It then computes a per-line delta and writes both line-level and run-level totals back to Airtable.

Main conclusion: this is not a parent sync flow like the WebSocket handler or the Downloader. It is a dedicated accrual reconciliation workflow centered on Media / Prod New and the two sys mapping tables.

5. Calculation logic

5.1 Per-record matching

For each loaded Media / Prod New record, the main Code node creates two matched subsets:

  • fakturamapping subset: rows whose ID Mediakop/Production equals the Media / Prod row's Sys. ID
  • orderallocationmapping subset: rows whose ID Mediakop or ID Production equals the same Sys. ID

5.2 Core numeric totals

Upplupna AutomationtotalBeloppExMoms - totalBriefCustomerSpendAllocated

Observed formula

The workflow explicitly calculates the main delta as actual mapped supplier cost minus allocated invoiced amount.

5.3 Additional line-level outputs

  • Upplupna Automation Up Until - copied from the run control record
  • Upplupna Automation Run Date - stamped with current execution time
  • Upplupna Automation Found Mapping Cost - raw fakturamapping subtotal
  • Upplupna Automation Found Allocations Invoiced - raw orderallocationmapping subtotal
  • Upplupa Comment - set to Has multiple allocation dates when a matched allocation row carries multiple invoice dates

5.4 Supplier-specific branch

The code contains a dedicated supplier subtotal routine for supplier ID GBG and supplier name 912609676-Adgrabber Goteborg AB (From Fortnox). This computes separate supplier-specific totals for mapping cost, allocated invoiced amount, and the resulting delta.

This branch is hardcoded. The workflow is therefore partly generic and partly bespoke: all rows are processed generally, but one supplier receives additional dedicated subtotals.

The active Airtable Update Media/Prod node writes the following fields back to each matched Media / Prod record:

6.2 Run-level aggregation

After batch processing is exhausted, the Total node sums the emitted line-level records and produces run-level totals:

  • Total Upplupna
  • Total Media/Prod New
  • Total Fakturamapping
  • Total Orderallocationmapping
  • Total Found Mapping Cost
  • Total Found Allocations Invoiced
  • supplier-specific GBG totals

6.3 Final control-record update

The final Airtable Update Automation node writes the aggregated totals back to the selected run record in the Automation Upplupna table.

7. Confirmed risks and quirks

7.1 Status spelling mismatch

Confirmed issue: the control-table schema embedded in the workflow shows status option Finshed (misspelled), while the Total code sets final status to Finished.

That mismatch is directly visible in the export. What cannot be proven from the export alone is whether the live Airtable field has already been corrected. But the inconsistency inside the workflow export is real and should be treated as a likely bug or stale-schema warning.

7.2 Legacy / disconnected nodes remain in the workflow

The export still contains old nodes such as Airtable Find Media/Prod1, Airtable Find Media/Prod2, Code1-V1, and Code -Test. These are not part of the active execution chain.

This suggests the workflow evolved in place and was not fully cleaned after iteration.

7.3 Broad active target scope

Because the active Media / Prod search only excludes Mgmt. Status = "Stangd", the run recalculates across all open line items rather than a narrowly date-scoped subset. This may be intentional, but it is broader than the old alternative nodes imply.

7.4 No explicit visible error branch

The control table supports an Error status option, but the visible active path in this export only clearly sets In progress at the beginning and a finished-state value at the end. No dedicated catch branch is visible in the main graph.

8. Interpretation of business purpose

Most likely business role

This workflow appears to be the project's automated accrual reconciliation pass for campaign line items. It compares actual supplier-side cost already attributed through sys.fakturamapping with already-invoiced allocation-side value coming from sys.orderallocationmapping, then persists the difference back to Media / Prod New and the run table.

That interpretation aligns with the uploaded Airtable reference docs, which describe the same table cluster as the accrual layer and describe UPPLUPNA as the difference between allocated spend and actual invoiced cost. The workflow export itself confirms the exact implementation currently used by n8n: the active code computes actual mapped cost minus allocated invoiced amount.

Important nuance: some older documentation phrases the concept as allocated spend minus invoiced cost, while this workflow's active code calculates cost minus allocated invoiced amount. That does not mean the workflow is wrong, but it does mean sign convention should be checked carefully when interpreting positive and negative values in reporting.

9. Final verdict

Verdict: Upplupna Calculation is an active, standalone Airtable-side calculation workflow. It is not a parent sync flow. Its purpose is to recalculate per-line and per-run accrual deltas across Media / Prod New using the two sys mapping tables, then write those values back for reporting and operational review.
