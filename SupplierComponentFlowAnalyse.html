2. Architectural Role and Dependencies

This workflow is narrow but important in the supplier-side branch of the sync graph. It does not sync supplier invoices itself. Instead, it ensures that one Fortnox supplier master record exists in Airtable so dependent workflows can safely attach a FortnoxSupplier link to supplier-invoice records.

DirectionWorkflowPayloadPurpose
Parent -> this component Downloader / WebSocket Handler / Supplier Invoice Component { id: <Fortnox supplier number> } Tells the component which single Fortnox supplier to fetch and normalize.
This component -> Credentials Workflow 109 No business payload Loads shared config such as Airtable base id, supplier table id, and the Fortnox ratelimit proxy id.
This component -> child workflows None None This component is a leaf node in the sync graph. It does not call Supplier Invoice File, Order, Invoice, or Article components.
This component -> downstream parent logic Supplier Invoice Component { id, should_sync } Supplies the Airtable supplier record id and a boolean gate that the supplier-invoice flow uses to decide whether to continue syncing that invoice.
Role relative to the wider model: the FortnoxSupplier table contains both Fortnox-synced suppliers and manual platform suppliers. This component only handles the Fortnox side. It cannot create the manual platform-only supplier records described in the field audit because it always starts from GET /3/suppliers/{id} in Fortnox.

3. Observed Execution Path

The workflow is structurally simple. It fetches one supplier from Fortnox, finds the matching Airtable supplier by SupplierNumber, lightly normalizes one field, then upserts the header record and returns a minimal output contract.

Parent sends one supplier id
  -> Credentials (workflow 109)
  -> GET /3/suppliers/{id} via ratelimit proxy
  -> Find Supplier by returned SupplierNumber
  -> extract Supplier object
  -> normalize WorkPlace so empty becomes null
  -> if Airtable supplier exists: update
  -> else: create
  -> return Airtable record id and should_sync boolean
1
Load shared config and fetch the Fortnox supplier
The workflow calls the shared credentials workflow, then issues GET /3/suppliers/{id} through the shared ratelimit proxy.
2
Find the existing Airtable supplier by SupplierNumber
It searches the Airtable FortnoxSupplier table with SupplierNumber = Fortnox.Supplier.SupplierNumber.
3
Extract the Supplier object from the HTTP envelope
The JSON Supplier node unwraps the Supplier object so downstream nodes work with the normalized Fortnox payload rather than the full HTTP response shell.
4
Normalize WorkPlace
The Code node sets WorkPlace = null whenever the value is missing or empty. No other visible field transformation happens here.
5
Upsert the supplier header
The Supplier Exists IF node decides between Update Supplier and Create Supplier based on whether the Airtable lookup returned a record.
6
Return a minimal result for parent workflows
The workflow ends by outputting only { id: <airtable record id>, should_sync: <boolean> } so parent workflows can continue chaining without carrying the full supplier payload.

4. Header Record Write Behavior

The write-side logic is straightforward. The main insight is not a long transformation pipeline, but rather which identity key it uses and how small the normalization layer actually is.

Observed elementWhat it doesWhy it matters
Find Supplier Looks up an existing Airtable supplier by SupplierNumber. That makes SupplierNumber the business identity used inside this n8n component.
JSON Supplier Extracts the inner Supplier object from the Fortnox API response. Downstream nodes write a clean supplier payload rather than the full envelope.
Code node Normalizes WorkPlace to null when missing. This suggests Airtable expects a proper empty numeric value rather than an undefined or empty-string payload for that field.
Supplier Exists IF node Tests whether the Airtable list result contains an existing supplier record. This is the upsert gate between update and create behavior.
Update Supplier / Create Supplier Writes the parent record into the Airtable FortnoxSupplier table. The supplier becomes available as a link target for supplier invoices and supplier-level rollups.
Supplier Component Output Returns the Airtable record id plus a computed should_sync boolean. The parent does not need the whole supplier object. It mainly needs a link id and a continue/stop signal for downstream invoice syncing.

Important design fact

The workflow does not use @url directly as the upsert key in the n8n component itself. The uploaded component finds records by SupplierNumber. That differs from the broader table-level audit language that describes the general supplier sync model using @url as the upsert key.

Likely reason for the design

Using SupplierNumber inside the component is operationally sensible because parent workflows already pass supplier ids in that form, and supplier invoices also refer to suppliers through SupplierNumber. The current source set does not explicitly justify the design, but it fits the dependency graph well.

5. Returned Output and Supplier-Invoice Gate

The most distinctive behavior in this component is the fact that it returns more than just an Airtable id. It also computes a boolean that downstream workflows use as an execution gate.

Returned fieldObserved expressionMeaning
id {{ $json["id"] }} The Airtable record id of the created or updated FortnoxSupplier record.
should_sync {{ $json["fields"]["Internal Status"] !== "Do not sync" }} A boolean derived from the Airtable supplier record after write. Parent workflows can use it to decide whether supplier-invoice sync should continue.
Confirmed downstream use: the uploaded Supplier Invoice Component calls this workflow, then immediately branches on should_sync. If the value is true, it continues writing the invoice. If false, it exits early instead of syncing that supplier invoice.

What this means in practice

The supplier component is not just a mirror flow. It also acts as the policy checkpoint that exposes whether this supplier is currently allowed into the supplier-invoice sync path.

Why this is likely placed here

This is a sensible location for the gate because the supplier record is the authoritative home of Internal Status. Once the component has ensured the supplier exists in Airtable, returning the gate decision alongside the supplier id avoids a second lookup in the parent workflow.

7. Confirmed Quirks and Unresolved Questions

7.1 The export is inactive, but the component is still referenced elsewhere

The uploaded JSON proves "active": false. Separately, the broader source set shows workflow 112 referenced by the Downloader, the WebSocket Handler, and the Supplier Invoice Component. That means the component remains part of the documented dependency graph even though this export was disabled when captured.

7.2 The component performs very little normalization

Compared with other child components, this one is sparse. The visible normalization logic only handles WorkPlace by forcing empty values to null. Everything else appears to pass through from the Fortnox supplier object into Airtable without additional type coercion in this component.

7.3 There is no explicit Fortnox-error fallback branch

Unlike the Customer Component, this workflow shows no separate recovery path when the Fortnox HTTP request fails. No continueErrorOutput rescue branch is visible in the uploaded export. If the supplier fetch fails, the workflow appears more likely to fail hard instead of returning a cached Airtable supplier id.

7.4 The sync gate is computed from Airtable, not from the Fortnox payload

The returned should_sync value is based on the Airtable field Internal Status, which the field audit describes as a manual operational field controlled by staff. This is important because it means the decision to continue supplier-invoice syncing is an Airtable-side policy decision, not a Fortnox-side one.

Still unresolved: the available sources leave one subtle ambiguity. The field audit says suppliers with Internal Status = "Do not sync" or with no status set are skipped by invoice sync. But the component's actual expression is only Internal Status !== "Do not sync", which would evaluate to true for blank or missing values. The current source set does not prove whether live supplier records always have a populated status, whether another upstream condition prevents blank records from reaching this point, or whether the field-audit summary is slightly stricter than the current n8n logic.

7.5 This workflow is only for Fortnox-backed suppliers

The field audit documents a second supplier type: manual platform suppliers marked Supplier type = "Platform - Not Fortnox". This workflow cannot be the creator of those records, because it always begins by fetching a Fortnox supplier endpoint. So the mixed FortnoxSupplier table should not be mistaken for evidence that this component manages all supplier records in the table.

7.6 Why the component returns should_sync instead of making the final decision itself

The most plausible reason is composability. The supplier component's job is to resolve the supplier master record and expose supplier policy. The Supplier Invoice Component then decides whether to continue or exit. That keeps the supplier master-data flow reusable while letting the invoice flow own the final branch structure.

8. Plain-English Verdict

This workflow is the supplier mirror-and-gate child flow for Fortnox suppliers.

Its core job is simple: fetch one supplier from Fortnox, upsert the matching FortnoxSupplier record in Airtable, and return the Airtable supplier id. But it also does one strategically important extra thing: it returns a should_sync flag derived from the supplier's Airtable Internal Status, which downstream invoice logic uses as the go / no-go gate for supplier-invoice syncing.

So, in practical terms, this is not just a passive mirror. It is the place where supplier identity becomes available in Airtable and where the supplier-side sync policy gets exposed to the rest of the Fortnox workflow graph.
