TABLE FortnoxOrder — Full Field Audit

Table ID: tblhNdAzxbWeNgEgW  |  Base: Betaportal Adgrabber 2.0 (appk463NGeDy17j9T)  |  Total fields: 231  |  Audit date: 2026-03-25

FortnoxOrder is the central table of the entire order management workflow. Unlike FortnoxCustomer (which is primarily a sync mirror), this table has two co-equal purposes: (1) a Fortnox sync table — n8n polls GET /3/orders and upserts order master data from Fortnox; and (2) an Airtable-native operational layer — allocation tracking, invoice reconciliation, campaign linking, QA checks, and budget vs. spend reporting. The Fortnox API fields account for roughly 78 of the 231 fields; the remaining 153 are Airtable-native and actively used.

Key architecture note — 3-level chain: Fortnox returns order row details as a nested JSON blob (OrderRows). n8n stores that blob and also parses each row into a separate child record in the Fortnox OrderRow table (tbl4kQjfGdqF1TWkS). Each Fortnox OrderRow record is then linked to one or more records in sys.orderallocationmapping (tblvru6tOgpB5yVQG), which maps the order row to a specific media buy or production brief in Media / Prod New (tblTf9sNLy464khq5). The full chain:

FortnoxOrder → Fortnox OrderRow → sys.orderallocationmapping → Media / Prod New (Mediaköp or Production)

Most of the rollup complexity in FortnoxOrder is aggregating budget, allocation, and spend values back up this chain. The link field in FortnoxOrder is named "Line Items" but it points to the Fortnox OrderRow table. The link field in Fortnox OrderRow is named "Line Item Allocation" but it points to the sys.orderallocationmapping table.

A — Fortnox-Side Fields  Origin: Fortnox / n8n sync

n8n polls GET /3/orders (and GET /3/orders/{DocumentNumber} for detail) and upserts records using @url as the upsert key. Any manual edit to an A-field in Airtable will be silently overwritten on the next sync. Changes to order data must be made in Fortnox first.

Boolean fields: Fortnox returns booleans (Cancelled, HouseWork, NotCompleted, CopyRemarks, WarehouseReady, VATIncluded, Sent) as JSON booleans but n8n stores them as the string "true" or "false" in Airtable multilineText or singleSelect fields. OrderRows is the full Fortnox line-item array serialized as a JSON string. EmailInformation and Labels are also stored as JSON strings.

Summary: Key Architectural Points

  • This is the central hub table. FortnoxOrder sits at the intersection of Fortnox sync, Fortnox OrderRow allocation, campaigns (Kampanj), invoicing (FortnoxInvoice), and customer management (Kunder). 14 views serve 14 different use cases.
  • OrderRows JSON blob vs. Fortnox OrderRow table. Fortnox returns line items as nested JSON in OrderRows. n8n stores that blob as-is (audit trail) and also parses each element into a separate record in the Fortnox OrderRow table (tbl4kQjfGdqF1TWkS), linked via the field named "Line Items".
  • 3-level allocation chain. FortnoxOrder → Fortnox OrderRow (link field: "Line Items") → sys.orderallocationmapping (link field: "Line Item Allocation") → Media / Prod New (Mediaköp or Production records). Most rollup complexity is aggregating values back up this chain. The confusing naming — link fields use "Line Items" and "Line Item Allocation" while the actual tables are named differently — is a known source of confusion.
  • Boolean storage pattern. 7 Fortnox boolean fields are stored as "true"/"false" strings. Formulas and automations must use string comparison (= "true"), not checkbox logic.
  • 9 confirmed dead fields (4 invalid rollups + 3 dependent invalid formulas + 2 dated one-offs). Safe to delete without affecting any active workflow.
  • No unused CRM layer. Unlike FortnoxCustomer (46 empty CRM fields), every category in FortnoxOrder is actively populated and in use.
