2. Architectural Role and Dependencies

This workflow is narrow and foundational. It does not sync orders, invoices, or allocations itself. Its only job is to make sure that one Fortnox article master record exists in Airtable so that downstream row-level workflows can link to it safely.

DirectionWorkflowPayloadPurpose
Parent -> this component Downloader / WebSocket Handler / Invoice Component / Order Component { id: <Fortnox ArticleNumber> } Tells the component which single Fortnox article to fetch and normalize into an Airtable link target.
This component -> Credentials Workflow 109 No business payload Loads shared config such as Airtable base id, article table id, and the Fortnox ratelimit proxy id.
This component -> child workflows None None This component is a leaf node in the sync graph. It does not call Customer, Order, Invoice, Supplier, or file workflows.
Role relative to the wider model: the uploaded schema notes describe FortnoxArticle as the product/article catalog in Airtable, linked to sys.mediatyp for media-type classification and used as a reusable master-data anchor from row-level tables. The component therefore exists primarily to keep that master-data mirror available.

3. Observed Execution Path

The visible execution path is deliberately simple. Compared with the Order and Invoice components, this workflow has no child-row reconciliation and no secondary branching beyond the update/create upsert gate.

Parent sends one article id
  -> Credentials (workflow 109)
  -> GET /3/articles/{id} via ratelimit proxy
  -> find Airtable FortnoxArticle by ArticleNumber
  -> extract Article object from HTTP payload
  -> pass payload through unchanged
  -> if article already exists: update Airtable record
  -> else: create Airtable record
  -> return Airtable article record id

4. Header Record Write Behavior

The write-side logic is minimal, but that minimalism itself is the key design feature of this workflow.

Observed elementWhat it doesWhy it matters
Find Article Looks up an existing Airtable article by ArticleNumber. That makes ArticleNumber the business identity used inside this n8n component.
JSON Article Extracts only the nested Fortnox Article object from the HTTP response. The Airtable write nodes receive a clean article-level payload rather than the full Fortnox wrapper structure.
Code node Returns the item unchanged. Unlike Customer, Order, and Invoice components, there is no visible normalization logic here. The article payload appears to be already compatible with the Airtable field layout.
Article Exists IF node Tests whether the Airtable list result contains an existing record. This is the upsert gate between update and create behavior.
Update Customer / Create Article Writes the parent record into the Airtable FortnoxArticle table. The article becomes available as a reusable link target for order rows and invoice rows.
Customer Component Output Returns only the Airtable record id. Parent workflows can immediately use the id for linking without carrying a large article object through later steps.

Important design fact

The workflow does not use @url directly as the lookup key in the n8n component itself. The uploaded component finds records by ArticleNumber. That differs from the field-audit language that describes the table-level upsert key as @url.

Likely reason for the design

Using ArticleNumber inside the component is operationally sensible because parent workflows pass article identifiers as plain Fortnox article numbers, and row-level records in orders and invoices also refer to articles by ArticleNumber. The current source set does not explicitly justify the choice, but it is consistent with the dependency graph.

5. Identity Model and Output Contract

This component is small enough that its most important behavior is not branching complexity but contract simplicity.

Output contract One field: { id: AirtableRecordId } The component is optimized for linking, not for downstream data transformation.
Normalization strategy Essentially none visible in the JSON export This suggests the Airtable schema was designed to mirror Fortnox article fields closely enough that generic pass-through writes are acceptable.
Failure handling No explicit fallback branch is visible If the Fortnox fetch fails, the component appears to have no alternate lookup-only rescue path like the Customer Component does.
Confirmed functional contract: this workflow is a reusable article id -> Airtable article record id resolver. That is why Order and Invoice row-sync flows can call it repeatedly without needing any extra payload shaping.
Failure boundary: the current JSON export does not show an onError = continueErrorOutput fallback branch or a direct Airtable rescue lookup. If the Fortnox fetch fails, this component appears more brittle than the Customer Component.

7. Confirmed Quirks and Unresolved Questions

7.1 The export is inactive, but the component is still referenced elsewhere

The uploaded JSON proves "active": false. Separately, the broader source set shows workflow 110 referenced as component_article by the WebSocket handler and by row-processing logic in Order and Invoice flows. That means the component remains part of the documented dependency graph even though this export was disabled when captured.

7.2 Two node names clearly come from a copy-pasted customer template

The update node is named Update Customer and the final output node is named Customer Component Output even though this is the Article Component. The runtime behavior still points to the article table, so this appears to be a labeling error rather than a logic error.

7.3 The article table behaves like a real master-data mirror in production

The field audit describes FortnoxArticle as a read-only Fortnox-to-Airtable mirror with complete API-field coverage, while the schema note describes it as the product/article catalog linked to internal media-type classification. That fits this component's very narrow master-data role.

7.4 The workflow uses ArticleNumber while the field audit describes @url as the upsert key

The available sources therefore document two identity concepts at once: a table-level canonical Fortnox URL and a workflow-level business lookup by article number. The current source set does not prove which one should be considered the primary identity across all historical versions of the integration.

Still unresolved: unlike the Customer Component, the current Article Component export does not show an explicit fallback path if Fortnox is temporarily unavailable. The available sources do not prove whether another error-handling layer compensates for that at runtime.

7.5 Why the minimal design is probably intentional

Articles are reference data, not transaction headers. Because the payload is flat and the parent workflows mainly need a link id, the simplest possible design is often the most robust: fetch one article, upsert it, return one id, and leave all transactional logic to OrderRow or InvoiceRow flows.

8. Plain-English Verdict

Bottom line: the Article Component is the smallest and cleanest Fortnox child component in the set. It behaves like a reusable master-data resolver: given one Fortnox article number, it guarantees one Airtable article record id.

That makes it strategically important even though it is small. Order-row and invoice-row workflows do not want to duplicate article-sync logic every time they encounter an ArticleNumber. Instead, they call this component, receive a stable Airtable id, and continue their own row upsert logic.

The most important caveats are not about business logic but about maintenance hygiene: two node names still refer to "Customer", the component's lookup key differs from the field-audit language, and there is no visible fallback path on fetch failure. None of those points change the core interpretation, but they are worth documenting for future cleanup.
