2. Architectural Role and Dependencies

This workflow is narrow but foundational. It does not sync invoices or orders itself. Instead, it ensures that one Fortnox customer master record exists in Airtable so that dependent workflows can safely attach links to it.

DirectionWorkflowPayloadPurpose
Parent -> this component Downloader / WebSocket Handler / Invoice Component / Order Component { id: <Fortnox customer number> } Tells the component which single Fortnox customer to fetch and normalize.
This component -> Credentials Workflow 109 No business payload Loads shared config such as Airtable base id, customer table id, and the Fortnox ratelimit proxy id.
This component -> child workflows None None This component is a leaf node in the sync graph. It does not call Order, Invoice, Article, or other components.
Role relative to the wider model: the FortnoxCustomer table is described in the field audit as a largely pure sync mirror in production. The workflow therefore acts as the identity anchor that lets orders and invoices link back to the correct Fortnox customer record in Airtable.

3. Observed Execution Path

The workflow is structurally simple, but it has one important twist: it contains a deliberate fallback branch for cases where the Fortnox API fetch fails.

Parent sends one customer id
  -> Credentials (workflow 109)
  -> GET /3/customers/{id} via ratelimit proxy
       - success output -> Find Customer by returned CustomerNumber
       - error output   -> Find Customer1 by Start.id in Airtable
  -> on success path:
       - extract Customer object
       - normalize selected fields for Airtable compatibility
       - if Airtable customer exists: update
       - else: create
       - return Airtable record id
  -> on error path:
       - do not write to FortnoxCustomer
       - look up an already-existing Airtable customer by CustomerNumber
       - return Airtable record id if found
1
Load shared config and fetch the Fortnox customer
The workflow calls the shared credentials workflow, then issues GET /3/customers/{id} through the shared ratelimit proxy.
2
Branch on HTTP success vs HTTP error
The HTTP node is configured with onError = continueErrorOutput, so the workflow has both a normal success branch and an explicit error-output branch.
3
On success, find by CustomerNumber
It searches the Airtable FortnoxCustomer table with CustomerNumber = Fortnox.Customer.CustomerNumber.
4
Normalize selected fields
It converts booleans to strings, serializes nested Fortnox objects to JSON strings, and forces CustomerNumber into an integer before writing to Airtable.
5
Upsert the customer header
It checks whether the Airtable customer already exists. If yes, it updates that record. If no, it creates a new record.
6
Return the Airtable record id
The workflow ends by outputting only { id: <airtable record id> }, which makes it easy for parent workflows to chain dependencies.
7
On HTTP error, try Airtable-only recovery
If the Fortnox fetch fails, the workflow searches Airtable directly by Start.id and returns an existing customer id if one already exists, without performing an update.

4. Header Record Write Behavior

The write-side logic is straightforward, but the normalization code is important because it reveals how this component bridges Fortnox data types to the Airtable schema.

Observed elementWhat it doesWhy it matters
Find Customer Looks up an existing Airtable customer by CustomerNumber. That makes CustomerNumber the business identity used inside this n8n component.
Code node Converts Active and ShowPriceVATIncluded to string values, casts CustomerNumber to integer, and serializes DefaultDeliveryTypes and DefaultTemplates to JSON strings. This matches the field-audit description that Airtable stores those Fortnox values in simplified text-compatible forms.
Customer Exists IF node Tests whether the list result contains an existing Airtable record. This is the upsert gate between update and create behavior.
Update Customer / Create Customer Writes the parent record into the Airtable FortnoxCustomer table. The Fortnox customer becomes available as a link target for orders and invoices.
Customer Component Output Returns only the Airtable record id. Parent workflows do not need the full customer payload. They only need a stable Airtable link id.

Important design fact

The workflow does not use @url directly as the upsert key in the n8n component itself. The uploaded component finds records by CustomerNumber. That differs from broader table-level audit language that describes the general customer sync model using @url.

Likely reason for the design

Using CustomerNumber inside the component is operationally sensible because all parent workflows pass the Fortnox customer number as id, and related orders and invoices also reference customers by that number. The current source set does not explicitly justify the choice, but it is consistent with the dependency graph.

5. Fallback Path When Fortnox Fetch Fails

This is the most distinctive behavior in the workflow. Unlike a naive sync component, this one has an explicit recovery branch that still tries to satisfy parent workflows even when the Fortnox HTTP request fails.

StepObserved node(s)Meaning
1HTTP Fortnox CustomerConfigured with onError = continueErrorOutput, so runtime errors are not fatal by default.
2Find Customer1Searches Airtable directly by CustomerNumber = Start.id if the Fortnox fetch errors out.
3Customer Component Output1Returns an existing Airtable record id without updating any field values.
Confirmed recovery model: if Fortnox is temporarily unavailable, a parent workflow can still continue provided that the customer already exists in Airtable.

What the fallback definitely does not do

The error branch does not normalize data, does not update stale values, and does not create a missing customer. It is a lookup-only rescue path.

Failure boundary: if the Fortnox fetch fails and the customer does not already exist in Airtable, this workflow does not have a visible create-on-error path. In that case, the parent may receive no usable Airtable customer id.

Practical effect of the design

This pattern suggests the component was designed to protect chained workflows such as Invoice Component and Order Component from transient Fortnox outages. It prioritizes returning a usable Airtable link id over forcing a hard stop in every failure scenario.

7. Confirmed Quirks and Unresolved Questions

7.1 The export is inactive, but the component is still referenced elsewhere

The uploaded JSON proves "active": false. Separately, the broader source set shows workflow 111 referenced as component_customer by the Downloader, WebSocket Handler, and Invoice Component. That means the component remains part of the documented dependency graph even though this export was disabled when captured.

7.2 The FortnoxCustomer table behaves like a pure sync mirror in production

The field audit describes the table as originally designed for both sync and CRM use, but confirms that in practice the CRM layer is empty in both production and DEV. That means this component should be understood primarily as a master-data mirror, not as a CRM workflow.

7.3 The component returns only an Airtable id, not a normalized business object

This is intentional. The output contract is minimal because parent workflows mainly need a link id, not the full customer payload. That makes the component easy to reuse inside larger workflows.

7.4 The fallback branch is useful but asymmetric

The workflow can recover from Fortnox fetch failure only when the customer already exists in Airtable. It does not contain a visible strategy for creating or reconstructing a missing customer without a valid Fortnox response.

7.5 There is no visible logic here for maintaining the Kunder bridge

The FortnoxCustomer table has a documented link to the operational Kunder table, but the uploaded customer component contains no node that searches or writes Kunder. If that bridge is maintained automatically anywhere, it is outside this workflow.

7.6 Why the component probably prefers CustomerNumber over @url

Because every parent passes a customer number and every dependent table also refers to customers through that number, CustomerNumber is the simplest common key for chaining workflows. The broader sync architecture may still rely on @url in other contexts, especially in bulk polling logic.

7.7 Why the recovery branch probably exists

The most plausible reason is resilience. If a parent workflow already knows the customer should exist, letting it proceed with an existing Airtable record id is operationally better than failing every chained sync on a transient Fortnox problem. That rationale is not explicitly documented, but it fits the observed structure well.

8. Plain-English Verdict

This workflow is the customer master-record child flow for Fortnox customers.

  • It receives one Fortnox customer number from a parent workflow.
  • It fetches the current Fortnox customer payload and upserts the Airtable FortnoxCustomer record.
  • It normalizes a few Fortnox data types so they fit the Airtable schema.
  • It returns only the Airtable customer record id so parent workflows can link invoices and orders correctly.
  • It also includes a lookup-only fallback path for cases where Fortnox fetch fails but the customer already exists in Airtable.
Bottom line: the uploaded export proves that workflow 111 is designed as a Fortnox -> Airtable customer component with one especially important resilience feature: when the Fortnox API call fails, it can still return an existing Airtable customer id if that customer was already synced earlier. The main unresolved questions are outside the component itself, especially how the Kunder bridge is maintained and whether live bulk-sync logic uses @url, CustomerNumber, or both as the effective upsert identity.
