2. What It Actually Does

The workflow is very small. Both entry points go to the same Output node.

Execute Workflow Trigger
  -> Output

Manual Trigger
  -> Output

Observed behavior

  • It has no Fortnox API request node.
  • It has no Airtable read or write node.
  • It has no business logic branch.
  • It simply returns hard-coded string values from one Set node named Output.

So the right way to think about it is:

This is not an authentication workflow in the runtime sense.
It is a shared constants/config workflow.

4. How Other Workflows Use It

Other workflows call 109 first, then use its output in expressions.

Example usageWhat it means
$node["Credentials"].json["airtable_base"] Use the shared Airtable base id instead of hard-coding it in each workflow.
$node["Credentials"].json["airtable_supplier_invoice_table"] Use the shared supplier-invoice table id when reading or writing supplier invoices.
$node["Credentials"].json["component_supplier"] Call workflow 112 without hard-coding 112 directly in the parent logic.
$node["Credentials"].json["component_supplier_invoice_file"] Let workflow 113 call file helper 120.
https://ratelimit.offertizer.com/{{ $node["Credentials"].json["ratelimit_id"] }} Build the proxy URL used for Fortnox API requests.
Practical meaning: instead of copying base ids, table ids, and child workflow ids into every workflow, the system keeps them in one simple place and reuses them everywhere.

Typical dependency pattern

Parent workflow starts
  -> call workflow 109 "Credentials"
  -> receive config JSON
  -> use config JSON in Airtable nodes, proxy URLs, and Execute Workflow nodes
  -> continue normal business logic

That is why so many other workflows have an early node named Credentials pointing to workflow 109.

5. What It Does Not Do

Important boundary: this workflow does not contain the live Fortnox OAuth bearer token, does not refresh OAuth, and does not grant Airtable access by itself.
  • The real Fortnox OAuth authentication still comes from attached n8n credentials such as Fortnox Oauth2.
  • The real Airtable authentication still comes from attached Airtable token credentials on Airtable nodes.
  • Workflow 109 only gives ids and references.

Simple distinction

Workflow 109 = config values
n8n credentials = real authentication objects

6. Plain-English Verdict

This workflow is the shared config dictionary for the Fortnox integration.

  • It does not sync customers, suppliers, orders, invoices, or files.
  • It does not talk to Fortnox or Airtable directly.
  • It returns one fixed JSON object containing ids that the rest of the workflows need.
  • That is why almost every major workflow depends on it first.
Bottom line: workflow 109 is a very small but central utility. It keeps base ids, table ids, child workflow ids, and the ratelimit proxy id in one place, so the rest of the Fortnox flows can stay consistent and easier to maintain.

End of workflow note.
