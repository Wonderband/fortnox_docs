Missing Flow Blueprint — PDF Webhook Server

How to build the 16th N8N flow that serves PDF files from disk to Airtable · Reference guide for migration

Context

Why This Flow Exists

When the SI File Component (#120) downloads a supplier invoice PDF from Fortnox, it cannot upload binary data directly to Airtable. Instead it does this:

  1. Saves the PDF to disk: /home/ubuntu/files/invoice.pdf
  2. Tells Airtable: "here is a URL where you can fetch the file"
  3. Airtable's server makes its own HTTP GET request to that URL
  4. Whatever answers that URL must return the PDF binary
  5. Airtable stores the binary permanently as a proper attachment

This webhook flow is the thing that answers step 4. Without it, Airtable's GET request returns 404, the attachment field stays empty, and the PDF is silently lost.

Why it is missing from your exports
This flow likely exists on the Offertizer server but was never exported together with the 15 main flows. It is a short utility flow — easy to overlook. It must be recreated on any new server before PDF attachments will work.
The Flow

Flow Overview — 3 Nodes

① Webhook Trigger
Listens for GET from Airtable
→
② Read Binary File Reads PDF from disk
/home/ubuntu/files/…
→
③ Respond to Webhook Sends PDF binary
back to Airtable

That is the entire flow. Airtable calls it, gets the PDF, stores it. Nothing else happens here.

Node 1 of 3

Webhook — the trigger

①
Webhook
n8n-nodes-base.webhook
HTTP MethodGET
Pathserve-pdf  — or any name you choose. N8N generates the full UUID URL automatically after you save.
Response ModeUsing "Respond to Webhook" Node
This is critical — it tells N8N to wait for Node 3 before sending a response, so the binary can be included.
Binary DataLeave default (off). The binary is handled by Node 3.
After saving Node 1 — do this immediately
N8N will display the webhook URL. It looks like:

https://your-server:5678/webhook/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx

Copy this UUID. You will paste it into the SI File Component (#120) and into the rate limiter migration guide. This UUID is unique to your N8N instance and cannot be predicted in advance.

What the incoming request looks like

When Airtable calls this webhook it sends:

incoming request from Airtable
GET /webhook/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
    ?file=invoice.pdf
    &t=1713612345678   ← cache-buster timestamp, can be ignored

The ?file= query parameter tells the flow which filename to read from disk. Node 2 reads this value dynamically.

Node 2 of 3

Read Binary File — load PDF from disk

②
Read Binary File
n8n-nodes-base.readBinaryFile
File Path /home/ubuntu/files/{{ $json.query.file }}
$json.query.file reads the ?file= value from the incoming request.
So if Airtable calls ?file=invoice.pdf this resolves to /home/ubuntu/files/invoice.pdf
Property Name data
The binary will be stored under this key. Node 3 reads it by this name.
Why use a dynamic path instead of hardcoding invoice.pdf
The original Offertizer flow always writes to invoice.pdf — one fixed filename. If two supplier invoices are processed within seconds of each other, the second PDF overwrites the first before Airtable has fetched it. By reading the filename from ?file= you open the door to fix this with unique filenames per invoice (e.g. ?file=12345.pdf). The dynamic path costs nothing here and makes the future fix trivial.
Node 3 of 3

Respond to Webhook — send the PDF back

③
Respond to Webhook
n8n-nodes-base.respondToWebhook
Respond WithBinary
Binary Propertydata  — must match the Property Name set in Node 2
Response Code200
Response Headers Add one header:
Content-Type = application/pdf
This tells Airtable the response is a PDF, not plain text or JSON.
What happens after this node fires
Airtable receives the binary PDF over HTTP. Airtable stores it permanently on its own servers and displays it as a native attachment in the supplier invoice record. The file on your server disk (/home/ubuntu/files/invoice.pdf) is no longer needed after this point — Airtable has its own copy.
After Creating the Flow

What to Update in SI File Component (#120)

Open the Code2 node inside SI File Component. Find the line that builds the Airtable attachment URL and replace the old Offertizer webhook URL with your new one:

Code2 node — before (old Offertizer URL)
return [{
  'Invoice PDF': [{
    filename: jsonPlus.Name,
    url: 'https://automation.offertizer.com/webhook/daa58b95-f6e5-42ea-b5c9-0efe21eb590d'
         + '?file=invoice.pdf&t=' + Date.now()
  }]
}]
Code2 node — after (your new server)
return [{
  'Invoice PDF': [{
    filename: jsonPlus.Name,
    url: 'https://YOUR-SERVER-DOMAIN:5678/webhook/YOUR-NEW-UUID'
         + '?file=invoice.pdf&t=' + Date.now()
  }]
}]
Two things to substitute
YOUR-SERVER-DOMAIN — the public hostname or IP of your new server. Must be reachable from the internet (Airtable's servers need to reach it). If N8N runs behind a reverse proxy (nginx/Caddy), use port 443 and your domain name instead of 5678.

YOUR-NEW-UUID — the webhook UUID that N8N generated when you saved Node 1 of this flow.

Optional improvement — fix the race condition at the same time

While you are editing Code2, also change the filename from the hardcoded invoice.pdf to the actual invoice number. This prevents two simultaneous downloads from overwriting each other.

Code2 node — with race condition fix
const invoiceId = $('Start').item.json.id  // e.g. "12345"

return [{
  'Invoice PDF': [{
    filename: jsonPlus.Name,
    url: 'https://YOUR-SERVER-DOMAIN:5678/webhook/YOUR-NEW-UUID'
         + `?file=${invoiceId}.pdf&t=` + Date.now()
  }]
}]

And in the Write Binary File node of SI File Component, change the path from:

/home/ubuntu/files/invoice.pdf

to:

/home/ubuntu/files/{{ $('Start').item.json.id }}.pdf
Node 2 of the webhook flow needs no change
Because Node 2 already uses {{ $json.query.file }} dynamically, it will automatically read 12345.pdf when Airtable calls ?file=12345.pdf. No edit needed there.
Testing

How to Test Before Relying On It

Step 1 — put a test PDF on disk

terminal
# Any PDF will do — copy one you have, or create a dummy
mkdir -p /home/ubuntu/files
cp /any/existing/file.pdf /home/ubuntu/files/invoice.pdf

Step 2 — activate the webhook flow in N8N

Open the flow in N8N and click the Active toggle in the top right. An inactive flow will not respond to requests.

Step 3 — call the webhook exactly like Airtable would

terminal
# Replace with your actual server and UUID
curl -o /tmp/test_result.pdf \
  "https://YOUR-SERVER:5678/webhook/YOUR-UUID?file=invoice.pdf"

# Check the result is a real PDF
file /tmp/test_result.pdf

Expected output:

/tmp/test_result.pdf: PDF document, version 1.x
If curl returns HTML or an error instead of a PDF
  • Check the webhook flow is Active in N8N (not just saved — it must be toggled on)
  • Check the UUID in your curl command matches exactly what N8N shows in the Webhook node
  • Check the file exists at the path: ls -la /home/ubuntu/files/invoice.pdf
  • Check N8N logs for execution errors: docker logs n8n or pm2 logs n8n

Step 4 — end-to-end test with a real supplier invoice

Trigger the SI File Component (#120) manually for a supplier invoice that has a PDF attached in Fortnox. Then open that invoice record in Airtable and verify the Invoice PDF attachment field contains a downloadable PDF — not just a broken link.

  • Add Node 1 — Webhook · Method: GET · Path: serve-pdf · Response Mode: Using "Respond to Webhook" Node

  • Connect the nodes: Webhook → Read Binary File → Respond to Webhook
  • Activate the flow (toggle it ON in N8N)
  • Update Code2 node in SI File Component (#120) — replace the Offertizer webhook URL with your new UUID

  • Run an end-to-end test — trigger #120 for a real invoice and check the Airtable attachment field
