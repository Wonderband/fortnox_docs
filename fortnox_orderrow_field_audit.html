TABLE Fortnox OrderRow — Full Field Audit

Table ID: tbl4kQjfGdqF1TWkS  |  Base: Betaportal Adgrabber 2.0 (appk463NGeDy17j9T)  |  Total fields: ~159  |  Audit date: 2026-03-26

Fortnox OrderRow is a hybrid table — it has two co-equal purposes: (1) a Fortnox sync table — n8n parses the OrderRows JSON blob from each FortnoxOrder record and upserts one record per row using RowId as the key; and (2) an Airtable-native operational layer — pricing adjustments, margin, allocation tracking, QA checks, and soft-deletion logic are all managed here by the team. Roughly 23 fields are Fortnox-side; the remaining ~136 are Airtable-native.

Position in the allocation chain: FortnoxOrder → Fortnox OrderRow → sys.orderallocationmapping → Media / Prod New

Each Fortnox OrderRow record represents one line item from a customer order. The FIELD Line Item Allocation (confusingly named — it points to TABLE sys.orderallocationmapping, not a field) distributes the row's value across one or more campaign records in Media / Prod New, split by month and allocation type. The full revenue side of P&L flows through this link.

A — Fortnox-Side Fields  Origin: Fortnox / n8n sync

n8n parses the OrderRows JSON array from each FortnoxOrder and upserts one record per row using RowId as the upsert key (no @url field on this table — RowId is the unique Fortnox-assigned identifier per row per order). LineItem Source = "Fortnox" on all live records confirms the Airtable-users push path has never been used.

String vs number types for Fortnox values: Several fields that are numbers in the Fortnox API payload are stored as singleLineText in Airtable (e.g. ContributionPercent, ContributionValue, DeliveredQuantity, OrderedQuantity). Formula fields (Delivered_QTY_integer, Ordered_QTY_integer copy) convert these to numbers using VALUE() for use in calculations.
