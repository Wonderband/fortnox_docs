Adgrabber Fortnox Invoice Component — Workflow Analysis

Workflow export reviewed: Adgrabber Fortnox Invoice Component v1.0.0  |  Analysis date: 2026-04-08  |  Workflow ID: 115

Scope of this note. The invoice component processes one invoice at a time. That does not imply that a customer has only one order or one invoice. This note separates the workflow-level handoff logic from the business data relationships, and it shows exactly which child components are called, what data is passed to them, what comes back, and what this component writes to Airtable.
Primary inspected sources: Adgrabber_Fortnox_Invoice_Component_v1_0_0.json, Adgrabber_Fortnox_WebSocket_Handler_v3_0_0.json, Adgrabber_Fortnox_Downloader_V3_0_0.json, Fortnox_Adgrabber_Credentials.json, fortnox_invoice_field_audit.html, and customer_field_audit.html.

2. What the Parent Passes Into This Component

The invoice component expects a very small input contract from its parent: an object with one field, id.

Parent flowHow it builds the inputMeaning of id
WebSocket Handler (116) For normal invoice events, it sets id = entityId. For payment-related invoice events, it sets id = payload.invoiceId. Fortnox invoice identifier used in GET /3/invoices/{id}.
Downloader (118) It sets id = DocumentNumber for each invoice record from the list API. Fortnox invoice document number, again used as the detail-fetch id. 
Parent -> Invoice Component
{
  "id": "2473"
}

Observed simplification

The parent does not send a full invoice payload. It sends only the invoice identifier. The invoice component then performs its own Fortnox detail fetch and does not rely on the parent for business data.

4. Step-by-Step Execution Path

Start
  -> Credentials (109)
  -> HTTP Fortnox API                     GET /3/invoices/{id}
  -> Customer Component Input            { id: Invoice.CustomerNumber }
  -> Customer Component Output           returns Airtable FortnoxCustomer record id
  -> Has No Order?
       -> yes: skip order child
       -> no: Order Component Input      { id: Invoice.OrderReference, customer_id: Customer Airtable id }
             -> Order Component Output   returns Airtable FortnoxOrder record id
  -> Find invoice in Airtable by DocumentNumber
  -> Normalize invoice payload
  -> Update existing invoice or create new invoice
  -> Store resulting Airtable invoice record id
  -> Should Sync rows?
       -> no: return invoice record id
       -> yes: loop InvoiceRows one by one
            -> Has No Article?
                 -> yes: skip article child
                 -> no: Article Component Input   { id: Row.ArticleNumber }
                       -> Article Component Output returns Airtable FortnoxArticle record id
            -> Find invoice row in Airtable by RowId
            -> Attach invoice link and optional article link
            -> Update existing row or create new row
            -> when last row is finished, return invoice record id

6. What Data Comes Back From Each Child Component

The child components are not returning full business payloads for this workflow to parse deeply. What this component really needs back is the Airtable record id of the synced related record.

Child componentValue the invoice component reads backHow it is used
Customer Component Outputid of the Airtable FortnoxCustomer recordWritten into the invoice header as FortnoxCustomer = [customerRecordId].
Order Component Outputid of the Airtable FortnoxOrder recordWritten into the invoice header as FortnoxOrder = [orderRecordId] if the order child ran.
Article Component Outputid of the Airtable FortnoxArticle recordWritten into the current invoice-row record as FortnoxArticle = [articleRecordId] if the article child ran.

Observed pattern

The invoice component uses child components primarily as relationship resolvers. It makes sure the related customer/order/article record exists first, then stores the returned Airtable id as a link on the invoice or invoice-row record.

7. What This Component Writes to Airtable

7.1 Invoice header table

After fetching and normalizing the Fortnox invoice header, the workflow upserts the FortnoxInvoice table by DocumentNumber.

Write targetHow it is foundWhat is written
FortnoxInvoice Find node filters Airtable by DocumentNumber Normalized Fortnox invoice payload, plus Airtable links to FortnoxCustomer and optionally FortnoxOrder, plus raw payload in Fortnox API Data.

The main code node explicitly sets:

  • FortnoxCustomer = [Customer Component Output.id] always
  • FortnoxOrder = [Order Component Output.id] only if an order child ran and returned successfully
  • serialized JSON fields such as InvoiceRows, EmailInformation, EDIInformation, and Labels
  • stringified booleans such as Booked, Cancelled, Sent, and VATIncluded

7.2 Invoice-row table

If the invoice has rows, the workflow upserts the FortnoxInvoiceRow table by RowId.

Write targetHow it is foundWhat is written
FortnoxInvoiceRow Find Row filters Airtable by RowId The current Fortnox invoice row payload, plus FortnoxInvoice = [invoiceRecordId], and optionally FortnoxArticle = [articleRecordId]. 
Code Row guarantees:
  FortnoxInvoice = [current Airtable invoice record id]
  FortnoxArticle = [current Airtable article record id]   only when article child ran successfully

8. Business Purpose of the Child Calls

Customer child call

Business purpose: every invoice belongs to a customer, so the Airtable customer record must exist before the invoice can be linked properly.

Practical effect: the invoice table remains connected to the customer table, and the customer table can maintain the inverse 1 customer -> many invoices relationship.

Order child call

Business purpose: many Fortnox invoices are generated from Fortnox orders, so this component tries to preserve that parent-order chain whenever OrderReference exists.

Practical effect: the invoice can link back to its parent order, which supports invoice-count rollups and revenue-chain analysis.

Article child call

Business purpose: each invoice row may refer to a catalog article. Syncing the article first lets the row link to the proper Airtable article record.

Practical effect: row-level analysis can reuse the shared Fortnox article dimension instead of leaving article references as raw text only.

9. What This Does Not Mean About Customers, Orders, and Invoices

This is where the earlier wording was easy to misread. The node name Has No Order means only: "this invoice record has no usable OrderReference". It does not mean that the customer has no orders overall.
LevelCorrect meaning
Workflow levelThis run processes one invoice. It may or may not call the order child depending on that invoice's OrderReference.
Invoice-record levelEach invoice links to one customer and usually one parent order.
Business-data levelOne customer can still have many orders and many invoices. The workflow does not contradict that.
Correct business relationships:
1 customer -> many orders
1 customer -> many invoices
1 order    -> many invoices   (possible)
1 invoice  -> many invoice rows

10. Status, Usage, and Caveats

QuestionAnswer
Is workflow 115 exported as active?No. The uploaded JSON marks it active: false.
Is it still architecturally important?Yes. The uploaded WebSocket Handler and Downloader both reference component_invoice, so this child remains part of the designed dependency graph.
Does the row sync always run?No. It runs only when the invoice payload contains at least one row.
Does the order child always run?No. It runs only when the invoice has a valid numeric OrderReference greater than 0.
Does the article child always run?No. It runs row-by-row only when the current row has a valid numeric/text article identifier that parses to a value greater than 0.

Operational reading

The design intent is to make the invoice component a self-sufficient sync unit for one invoice. A parent flow only needs to know the invoice id. This child workflow then resolves every required related record and writes both the header and the rows.
