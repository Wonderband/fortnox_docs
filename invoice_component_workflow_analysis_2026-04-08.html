Adgrabber Fortnox Invoice Component — Workflow Analysis

Workflow: Adgrabber Fortnox Invoice Component v1.0.0  |  Workflow ID: 115  |  Role: Child component in the Fortnox sync graph

Core role: This component syncs one Fortnox invoice at a time. It receives a single invoice identifier from a parent workflow, fetches the full invoice from Fortnox, ensures the related customer and parent order exist in Airtable, writes the invoice header record, and is designed to parse invoice rows into the dedicated FortnoxInvoiceRow table.
Status nuance: The uploaded JSON export of workflow 115 is marked inactive as a standalone workflow export. However, the broader uploaded architecture still references the invoice component from parent workflows. Therefore, this note treats the component as architecturally relevant even though the specific uploaded export is not an active top-level trigger workflow by itself.

Business purpose

The invoice component exists to normalize the customer-side invoice chain in Airtable. It does not only mirror invoice header fields; it also tries to ensure the invoice is connected to the correct customer, the correct parent order, and the correct line-level records and articles.

Need Why this component handles it
Customer linkage An invoice belongs to one Fortnox customer record. The component calls the Customer Component first so the customer exists before the invoice is written.
Order linkage If the invoice contains an OrderReference, the component ensures that parent order exists in Airtable, then links the invoice back to it.
Invoice header sync The component upserts the parent FortnoxInvoice record in Airtable.
Invoice row normalization The component is designed to explode InvoiceRows into child records in FortnoxInvoiceRow, one Airtable record per Fortnox row.
Article linkage When an invoice row contains an ArticleNumber, the component calls the Article Component so the row can link to a reusable FortnoxArticle master record.

Parent input and child dependencies

Direction Workflow Payload Purpose
Parent → this component WebSocket Handler / Downloader / similar parent { id: <invoice document number> } Tells the component which single invoice to fetch and process.
This component → Customer Component Workflow 111 { id: Invoice.CustomerNumber } Ensures the invoice customer exists in Airtable and returns the Airtable customer record id.
This component → Order Component Workflow 114 { id: Invoice.OrderReference, customer_id: AirtableCustomerId } Ensures the invoice's parent order exists in Airtable and returns the Airtable order record id.
This component → Article Component Workflow 110 { id: InvoiceRow.ArticleNumber } Ensures the row's article exists in Airtable and returns the Airtable article record id.

Explicit workflow schema

1
Receive parent input
The parent workflow passes only a single invoice identifier, typically as id.
This component processes one invoice at a time, not a full batch directly.
2
Fetch invoice from Fortnox
The component calls GET /3/invoices/{id} through the shared Fortnox ratelimit proxy and gets the full invoice payload, including invoice header fields and the InvoiceRows JSON array.
3
Resolve customer
It sends CustomerNumber to the Customer Component. That child either creates or updates the customer in Airtable and returns the Airtable customer record id.
4
Resolve parent order if possible
If OrderReference is a valid number, the component sends OrderReference and the returned Airtable customer id to the Order Component. That child either creates or updates the order in Airtable and returns the Airtable order record id.
5
Upsert invoice header
The component finds the invoice in Airtable by DocumentNumber and either updates the existing FortnoxInvoice record or creates a new one. It writes the returned customer/order ids into the invoice link fields.
6
Split invoice rows
The component iterates through the Fortnox InvoiceRows array, one row at a time.
7
Resolve article per row
If an invoice row contains a valid ArticleNumber, the component sends that number to the Article Component and receives an Airtable article record id in response.
8
Upsert invoice-row records
For each row, the component is designed to find a row by RowId and either update or create a record in FortnoxInvoiceRow, linking it back to the Airtable invoice record and, when available, the Airtable article record.

What Airtable tables may be touched by one run

Table Direct or indirect Expected action
FortnoxInvoice Direct Create or update one invoice header record.
FortnoxCustomer Indirect via Customer Component Create or update the related customer master record.
FortnoxOrder Indirect via Order Component Create or update the parent order, if OrderReference exists.
Fortnox OrderRow Indirect via Order Component May be updated because the Order Component itself parses and upserts order rows.
FortnoxArticle Indirect via Article Component Create or update article master records for invoice rows that contain valid article numbers.
FortnoxInvoiceRow Direct Designed to create or update one Airtable row record per Fortnox invoice row.

Data model clarification

Important relationship clarification: This component processes one invoice record at a time, but that does not mean a customer has only one invoice or only one order. The intended Airtable model is still:

1 customer → many orders
1 customer → many invoices
1 order → many invoices (possible)
1 invoice → many invoice rows

Observed contradiction: designed row sync vs empty production row table

What the design says

The invoice component JSON contains a clear row-processing path: it receives InvoiceRows, iterates through them, resolves articles, and is structured to create or update records in the dedicated FortnoxInvoiceRow table.

What production observations say

Production evidence says something different: the parent FortnoxInvoice table is populated and stores the raw InvoiceRows JSON blob, but the normalized child table FortnoxInvoiceRow is empty. In other words, row data clearly arrives inside the header payload, but the child-row table is still not materialized in production.

Current best statement: We can confidently say that the workflow is designed to populate FortnoxInvoiceRow, and we can also confidently say that the production table has remained empty in the audited observations. We cannot currently explain the exact runtime reason for this mismatch from the available sources alone.

Plausible explanations, but still only hypotheses

  • The live workflow version being executed is not identical to the uploaded export.
  • The invoice header part runs, but the child row-processing branch is bypassed or never reached in practice.
  • The parent workflows do call the invoice component, but not in a way that results in row-table persistence.
  • The row branch exists architecturally but is disabled, broken, or short-circuited in the live n8n setup.

These are hypotheses only. The real explanation has not yet been proven from the current evidence set.

Bottom-line interpretation

This component should be read as an invoice orchestration child workflow. It is not just a simple header mirror. Its intended role is:

  1. ensure dependencies exist (customer, maybe order, maybe articles)
  2. upsert the invoice header
  3. explode invoice rows into FortnoxInvoiceRow

However, the third step is currently contradicted by production observations. Therefore, the authoritative conclusion at this moment is:

Authoritative conclusion: The component is architecturally designed to populate FortnoxInvoiceRow, but the production table is still empty in the observed data. The exact operational reason for that mismatch is not yet discovered.
