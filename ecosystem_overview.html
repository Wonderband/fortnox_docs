Diagram 2 of 5  ·  Downloader #118 — Batch Sync (Fortnox → Airtable)
The Downloader runs every minute (triggered by Airtable Settings changes). It cycles through 6 entity families, fetching paginated lists from Fortnox and passing each ID to its dedicated component. Page counters are persisted in the Settings table so the download survives restarts. NOTE: passes sync_files: "no" to Supplier Invoice — PDFs are NOT downloaded during batch sync.

Diagram 3 of 5  ·  WebSocket Handler #116 — Real-time Sync
The WebSocket Handler stays permanently connected to Fortnox. When an entity changes in Fortnox, the event arrives here within seconds. The handler routes each event to the right component by topic. KEY DIFFERENCE vs Downloader: passes sync_files: "yes" to Supplier Invoice — so PDFs ARE downloaded for real-time events. Invoice payment events use a different field (payload.invoiceId) than all other events (entityId).

110 Article Component v1.0.0 Active

Purpose

Fetches a single article from Fortnox and upserts it into the Airtable Articles table.

Input

{ id: ArticleNumber }

Step-by-step

  1. Call Credentials (#109) to get config
  2. GET /3/articles/{id} via Offertizer rate-limit proxy
  3. Search Airtable Articles for matching ArticleNumber
  4. If found → UPDATE record; if not → CREATE record
  5. Return Airtable record ID

Output

{ airtable_id: "recXXXX" }

111 Customer Component v1.0.0 Active Note

Purpose

Fetches a customer from Fortnox and upserts into Airtable Customers table. Called by Invoice, Order, Downloader, WebSocket.

Input

{ id: CustomerNumber }

Data Transformations

  • Boolean fields (Active, ShowPriceVATIncluded) → stored as strings ("true"/"false")
  • Complex objects (DefaultDeliveryTypes, DefaultTemplates) → JSON-serialized strings

⚠️ Silent error handling

  • The HTTP Fortnox request has onError: continueErrorOutput. If Fortnox returns an error (e.g., customer not found), the flow continues without failing. The downstream Airtable write may receive empty data.
112 Supplier Component v1.0.0 Active

Purpose

Fetches a supplier from Fortnox and upserts into Airtable Suppliers table. Returns a should_sync flag used by Supplier Invoice Component.

Input

{ id: SupplierNumber }

Special Logic

Reads the Internal Status field on the Airtable supplier record. Returns should_sync: true/false based on this field. The Supplier Invoice Component (#113) uses this to decide whether to proceed with invoice sync for that supplier.

Output

{ airtable_id: "recXXXX", should_sync: true }

113 Supplier Invoice Component v1.0.0 Active Issues

Purpose

Fetches a supplier invoice from Fortnox, ensures the supplier exists in Airtable, and conditionally triggers PDF file download.

Input

{ id: GivenNumber, sync_files: "yes" | "no" }

Step-by-step

  1. Call Credentials (#109)
  2. GET /3/supplierinvoices/{GivenNumber}
  3. Call Supplier Component (#112) → get should_sync flag
  4. If should_sync = false: skip and return
  5. Upsert supplier invoice in Airtable
  6. If sync_files = "yes": call Supplier Invoice File Component (#120)

Data Transformations

  • Balance → string
  • Total, VAT → parsed as floats (stripped of "SEK" currency)
  • SupplierInvoiceRows, Vouchers → JSON-serialized strings

⚠️ File sync only on real-time events

  • Downloader (#118) calls with sync_files: "no"
  • WebSocket Handler (#116) calls with sync_files: "yes"
  • Result: PDFs are only downloaded for invoices that trigger a WebSocket event. Historical bulk-synced invoices will never have their files downloaded.
114 Order Component v1.0.0 Active

Purpose

Fetches a complete order (header + all rows) from Fortnox. Ensures customer and articles exist in Airtable. Rebuilds order rows on every sync.

Input

{ id: DocumentNumber, customer_id?: "recXXXX" }

Row Sync Logic

  • Fetches all existing Airtable OrderRows for this order
  • Marks ALL existing rows as Deleted
  • Then creates/updates rows from fresh Fortnox data
  • This means rows are logically "rebuilt" on each sync — old rows get the Deleted flag rather than being physically deleted

Internal Status Gate

Reads Internal Status field on the Airtable order record. If set to "Do not sync", the flow skips row processing.

Output

{ airtable_id: "recXXXX" }

115 Invoice Component v1.0.0 Active Issues

Purpose

The most complex component. Syncs an invoice with all linked entities (customer, order, article rows) to Airtable.

Input

{ id: DocumentNumber }

Step-by-step

  1. Call Credentials (#109)
  2. GET /3/invoices/{DocumentNumber}
  3. Call Customer Component (#111) with invoice's CustomerNumber
  4. If OrderReference ≠ 0: call Order Component (#114)
  5. Upsert invoice in Airtable Invoices table (linked to customer + order)
  6. For each row with an ArticleNumber: call Article Component (#110)
  7. Upsert each row in Airtable InvoiceRows table

Data Transformations

  • Boolean fields → strings
  • InvoiceRows, EmailInformation, EDIInformation, Labels → JSON-serialized strings stored on the parent invoice record

⚠️ Invoice Row Table Empty in Production

  • The flow is coded to write individual rows to the InvoiceRow table (tblDkNRTKMTpMTLK3)
  • In production, this table is reportedly empty
  • InvoiceRows ARE stored as a JSON string within the parent invoice record
  • Possible causes: (a) a branch condition prevents row writes, (b) all invoices have rows with no ArticleNumber (triggering the "Has No Article" skip path), (c) the Airtable write step has a bug, (d) the row table was never populated and is a planned-but-unfinished feature
  • Needs direct investigation in the running N8N instance.
116 WebSocket Handler v3.0.0 Active

Purpose

Listens on the Fortnox WebSocket for real-time entity change events and routes each event to the correct component flow.

Topics Subscribed

  • articles
  • invoices (incl. payment events)
  • orders
  • customers
  • suppliers
  • supplier-invoices

Event Routing

118 Downloader V3.0.0 Active

Purpose

Main batch orchestrator. Cycles through 6 Fortnox entity families (invoices, orders, customers, articles, suppliers, supplier invoices), fetches paginated lists, and delegates each item to its component flow.

State Machine (Airtable Settings table)

Uses the Settings table as a persistent state machine:

  • Download Invoices / Download Orders / etc. — boolean flags to enable each family
  • Invoices Downloader Page / Orders Downloader Page / Supplier Invoices Downloader Page — persisted page counters (survive restarts)
  • Customers, Articles, Suppliers: in-memory loop (no persistent page counter)

Pagination Logic

  1. Read current page from Settings
  2. Fetch one page from Fortnox API
  3. For each item on the page: call component (batchSize=1)
  4. If last page: reset page counter to 1
  5. Else: increment page counter in Settings
  6. Settings update triggers the next Downloader run (1-minute poll)

Key Parameters Passed to Components

  • Invoices → #115: { id: DocumentNumber }
  • Orders → #114: { id: DocumentNumber }
  • Customers → #111: { id: CustomerNumber }
  • Articles → #110: { id: ArticleNumber }
  • Suppliers → #112: { id: SupplierNumber }
  • Supplier Invoices → #113: { id: GivenNumber, sync_files: "no" }

— WebSocket/Downloader Reload (Watchdog) Active

Purpose

An ops watchdog flow — not a data sync flow. It does not touch Fortnox business data or Airtable directly. It keeps the two core long-running flows alive by cycling them via the N8N REST API.

Trigger

Schedule Trigger — fires every hour. Timezone: Europe/Stockholm. Immediately splits into two parallel branches on each fire.

Branch A — Unconditional restart of WebSocket Handler (#116)

  1. POST /api/v1/workflows/116/deactivate
  2. Wait 10 seconds
  3. POST /api/v1/workflows/116/activate
  4. Check response: if statusCode ≠ 200 → loop back to Wait and retry activation until 200
  5. On success: set Done = true

Rationale (inferred): WebSocket long-lived connections can silently stale — forcing a reconnect every hour keeps the Fortnox event stream fresh.

Branch B — Conditional restart of Downloader (#118)

  1. GET /api/v1/executions?workflowId=118&limit=1 — fetch latest execution
  2. Check: is stoppedAt older than now − 2 hours?
  3. If yes (stale): POST deactivate → wait 10 s → POST activate (no retry logic)
  4. If no (healthy): do nothing

Rationale (inferred): if the Downloader's Settings-driven state machine gets stuck, this flow detects staleness and kicks it back to life.

Asymmetry between the two branches

PropertyBranch A — #116 WebSocketBranch B — #118 Downloader
Restart conditionAlways (unconditional)Only if stoppedAt > 2 hours ago
Activation retryYes — loops until HTTP 200No — fire-and-forget
Deactivation checkNot checkedNot checked

⚠️ Known issues

  • Node is named "If Last Execution Hour Ago" but the actual expression uses $now.minus({ hours: 2 }) — 2-hour threshold, not 1 hour. Label is misleading for anyone reading the canvas.
  • Branch B activation has no retry: if /activate returns non-200, the restart silently fails.
  • Health check for #118 uses only stoppedAt age — does not inspect execution status or detect a currently running (but possibly frozen) instance.
