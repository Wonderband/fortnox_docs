Why this matters

The uploader is designed as a controlled push path. It does not scan the whole order table. It waits for a record to be queued by Airtable field state, then processes exactly that order.

5. Create vs. Update Logic

The workflow chooses between create and update by looking at the Airtable order field DocumentNumber.

ConditionSwitch outcomeFortnox API callObserved meaning
DocumentNumber empty Create branch POST https://api.fortnox.se/3/orders/ Create a brand-new Fortnox order from Airtable data.
DocumentNumber not empty Update branch PUT https://api.fortnox.se/3/orders/{DocumentNumber} Update the existing Fortnox order with the current Airtable data.
Confirmed create/update rule: the uploader is idempotent at the business-record level only through the Airtable DocumentNumber field. That field is the sole observed switch deciding whether the workflow creates a new Fortnox order or updates an existing one.

Likely behavioral implication

Because the update branch sends a full reconstructed Order body, not a small patch, the workflow appears intended to treat Airtable as the current source of truth for the Fortnox order header and row content. The uploaded JSON proves the full-body send; it does not by itself prove how Fortnox merges or replaces row-level changes internally.

6. Write-Back Behavior

After either the POST or PUT request succeeds, the workflow writes back to the same Airtable order record.

Write-back fieldObserved valueMeaning
Sync FortnoxnullClears the queue flag so the record is no longer marked for upload.
DocumentNumber$json["Order"]["DocumentNumber"]Stores the Fortnox order number returned by the API response.
Important operational fact: the Fortnox response becomes the source of the persisted order number. That means the create path not only creates the order in Fortnox, but also closes the Airtable-to-Fortnox loop by writing the returned DocumentNumber back into Airtable.

This workflow is much narrower than the downloader or WebSocket handler. It does not call any entity component workflows. Its only workflow dependency is the shared config workflow 109.

8. Confirmed Quirks and Risks

8.1 What is proven: the uploaded export is inactive

The workflow JSON shows "active": false. That is a direct fact from the uploaded n8n export, so the version documented in this note was inactive at the moment that export was taken.

8.2 What is also proven inside the uploaded workflow graph

No uploaded parent workflow in this source set is shown calling workflow 119 via executeWorkflow. Within the uploaded dependency graph, this workflow appears as a top-level trigger flow, not as a child workflow invoked by another parent.

8.3 Production Airtable base + sandbox Fortnox credential

The trigger points at production Airtable base appk463NGeDy17j9T, while the Fortnox HTTP nodes use credential Fortnox Sandbox OAuth2. That exact environment pairing is directly visible in the JSON export.

Why that environment mix matters: if this export reflects real runtime configuration rather than a staging snapshot, it would mean the workflow reads real Airtable production orders while writing to a sandbox Fortnox tenant. The uploaded files prove the credential names and IDs, but not whether those names are still semantically accurate today.

8.4 Empty-line-item orders may be allowed structurally but could be operationally risky

The workflow does not hard-stop when no linked order rows exist. It simply sends OrderRows = []. That may be intentional for draft headers, or it may be a weak validation point. The JSON confirms the empty-array path; it does not show whether Fortnox rejects, accepts, or warns on such payloads.

8.5 Current practical usage appears inactive, but that part is an inference

The uploaded export proves that this workflow was disabled when exported, and the uploaded workflow graph does not show any other parent flow calling it. Those are direct facts. A stronger sentence such as "this flow is not used right now in the live system" goes one step beyond the export itself. That broader operational conclusion is well-supported by the source set, but it is still an inference unless confirmed from the live n8n instance directly.

8.6 The uploader is probably a queue processor rather than a fully automatic sync mirror

Because the workflow requires Sync Fortnox = "To sync", it appears designed for explicit outbound publishing, not for every Airtable edit to auto-push immediately. That matches a controlled write-queue pattern rather than a passive bidirectional sync.

9. Plain-English Verdict

This workflow is the order publishing path from Airtable into Fortnox.

  • It watches the Airtable FortnoxOrder table for records explicitly flagged To sync.
  • It marks the order In progress, loads linked Airtable line items, and builds a Fortnox order payload.
  • It creates a new Fortnox order if DocumentNumber is empty, or updates the existing Fortnox order if DocumentNumber already exists.
  • On success, it clears the sync flag and writes the resulting Fortnox DocumentNumber back into Airtable.
Bottom line: this document proves that workflow 119 is an Airtable -> Fortnox push workflow for orders and that the uploaded export is inactive. The stronger statement that it is currently unused in the live system is a reasonable interpretation of the source set, but it should be read as an inference rather than as direct runtime proof from n8n.
