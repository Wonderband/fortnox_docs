3. Observed Execution Path

The workflow has two phases: discover the relevant file connection, then download and attach the file branch result to the existing Airtable invoice record.

Parent sends one supplier invoice number (GivenNumber)
  -> Credentials (workflow 109)
  -> initialize paging state { CurrentPage, Data[] }
  -> GET /3/supplierinvoicefileconnections?page=n via ratelimit proxy
  -> filter current page where SupplierInvoiceNumber == Start.id
  -> accumulate matches into Data[]
  -> if more pages remain: loop
  -> after last page: pick Data[0] or {}
  -> branch on "Has file"
       - false -> return { finished: true } and stop
       - true  -> continue
  -> call httpbin with OAuth credential to echo back live Authorization header
  -> GET /3/archive/{FileId} as binary stream
  -> write binary file to /home/ubuntu/files/invoice.pdf
  -> find Airtable supplier invoice by GivenNumber
  -> if Airtable record exists: build Invoice PDF payload
  -> update Airtable record
  -> return { downloaded: true }
1
Load shared config and initialize the paging loop
The workflow starts by calling the shared credentials workflow, then the first Code node creates a small paging state object containing CurrentPage and an accumulator array called Data.
2
Scan the Fortnox supplier-invoice file-connection endpoint
Each pass calls GET /3/supplierinvoicefileconnections?page=n through the shared ratelimit proxy.
3
Filter for the one supplier invoice number passed in by the parent
Code1 filters the current page down to rows where SupplierInvoiceNumber === Start.id, then appends the matches into the running Data array.
4
Loop until the last page, then keep only the first match
Not Last Page controls the paging loop. After the last page, JSONPlus converts Data[0] ?? {} into the working object for the rest of the flow.
5
Obtain a live Authorization header for the custom archive request
The workflow sends one request to https://httpbin.org/anything using the attached Fortnox OAuth credential, then reads the echoed Authorization header back inside custom code.
6
Download the archive file as binary
Code4 performs a direct request to /3/archive/{FileId} using that live Authorization header and stores the response in binary.data.
7
Write the binary locally on the n8n host
Write Binary File writes the stream to the local filesystem path /home/ubuntu/files/invoice.pdf. This is server-side storage on the machine or container running n8n.
8
Update the Airtable invoice record with a payload
The workflow finds the existing Airtable supplier invoice by GivenNumber, builds an Invoice PDF field value in Code2, and updates that existing record.
9
Return a small status object
The success path returns { downloaded: true }. The no-file path returns { finished: true }.

4. What the File Branch Actually Writes

The most important clarification from the later review is that this workflow performs two different writes, and they are not the same thing.

Write targetObserved nodeWhat is writtenMeaning
Local filesystem on the n8n hostWrite Binary File/home/ubuntu/files/invoice.pdfThe downloaded Fortnox archive stream is materialized as a local server file.
Airtable supplier invoice recordCode2 -> Airtable UpdateInvoice PDF = [{ filename, url }]The Airtable record is updated with an attachment-style field payload, not with the raw binary bytes.

What the payload looks like

{
  "Invoice PDF": [
    {
      "filename": "<invoice file name>.pdf",
      "url": "https://automation.offertizer.com/webhook/...?...file=invoice.pdf&t=<timestamp>"
    }
  ]
}

This is the structured field value written into Airtable. The workflow does not pass binary.data directly into the Airtable node.

Meaning of the local path: /home/ubuntu/files/invoice.pdf is a file path on the server or container where the n8n workflow runs. It is not Airtable storage and it is not Fortnox storage.
Important correction: the earlier artifact wording could be read as if workflow 120 directly “uploaded binary into Airtable”. The visible export does not prove that. What it proves is: Fortnox binary -> local file write -> Airtable update by URL payload.

5. What Is Proven vs. Unresolved

5.1 Proven from the workflow export

  • Fortnox archive download happens in custom code. The code requests /3/archive/{FileId} and stores the response as binary.
  • The binary is written locally. The file is written to /home/ubuntu/files/invoice.pdf.
  • The target Airtable record is found by GivenNumber. This helper assumes the supplier-invoice header already exists.
  • The value written into Airtable is an attachment-style payload. Code2 builds Invoice PDF = [{ filename, url }].
  • The URL in that payload is external. It points to automation.offertizer.com/webhook/..., not directly to a Fortnox archive URL.
  • The workflow uses httpbin as a token-echo workaround. It obtains the live OAuth Authorization header there and reuses it in custom code.

5.2 Not proven yet / still unresolved

  • The exact webhook implementation is not shown. The source set does not include the backend or workflow that serves the file behind the automation.offertizer.com/webhook/... URL.
  • Direct raw-binary upload into Airtable is not visible. No node in this export sends binary.data directly into Airtable.
  • The last mile of file delivery to Airtable remains unresolved. The export strongly suggests that Airtable is given a URL and later fetches the file through that URL, but the current source set does not show how that URL is implemented.
  • It is not proven whether the webhook serves the exact local file directly. It may map to the local file path, or it may proxy another service or storage layer.
Current conclusion: we have proved local binary download, local file write, and Airtable attachment-field update by URL payload; we have not yet proved the exact mechanism by which Airtable receives the actual binary bytes behind that URL.

8. Confirmed Quirks and Open Questions

8.1 The export is inactive, but the helper is still referenced by workflow 113

The uploaded JSON proves "active": false. Separately, the supplier-invoice component still references workflow 120 as component_supplier_invoice_file. That means the helper remains part of the documented dependency graph even though this captured export was inactive.

8.2 The workflow scans the entire connection list and only keeps the first match

The paging loop accumulates all matching connection rows into Data, but JSONPlus only uses Data[0]. If multiple Fortnox file connections exist for the same supplier invoice, the current export processes only the first one.

8.3 The helper uses httpbin as a token-extraction workaround

The workflow calls https://httpbin.org/anything with the Fortnox OAuth credential so the returned response includes the live Authorization header. The next code node reuses that header for the direct archive request. The shared workflow 109 provides config values such as ids and table names; it does not visibly expose the live Fortnox bearer token itself.

8.4 The helper writes to a fixed local filename before building the Airtable payload

The binary is always written to /home/ubuntu/files/invoice.pdf, while the later field payload uses a webhook URL with a cache-busting timestamp. The export proves this exact path and payload shape; it does not prove how concurrency, cleanup, or file serving are handled around that fixed filename.

8.5 The helper updates only when the Airtable header already exists

There is no visible create branch after Airtable Exist. So the file helper is structurally downstream of the header-sync workflow and cannot stand on its own as a complete supplier-invoice importer.

Still unresolved: the exact mechanism that makes the binary reachable to Airtable is outside the current source set. The workflow clearly writes a URL-based payload into the Invoice PDF field, but the webhook/backend that serves the file behind that URL is not included here.

8.6 Why the helper is split into its own workflow

Keeping file-download logic in a separate child workflow probably reduces complexity inside the main supplier-invoice component and makes file sync easy to suppress with the parent workflow's sync_files flag. The split is consistent with the broader componentized design of this Fortnox integration, although that rationale is inferred rather than explicitly documented inside the export.

9. Plain-English Verdict

This workflow is the PDF/file-enrichment helper for Fortnox-synced supplier invoices.

  • It receives one supplier invoice number from the parent supplier-invoice component.
  • It scans Fortnox file-connection pages until it finds a matching file connection for that invoice.
  • It downloads the first matching archive file and writes it to the n8n host filesystem.
  • It then looks up the already-existing Airtable supplier invoice by GivenNumber.
  • If that Airtable record exists, it updates the Invoice PDF field using a URL-based attachment payload and returns { downloaded: true }.
  • If no file exists, it exits cleanly with { finished: true }.
Bottom line: the uploaded export proves that workflow 120 is a Fortnox -> Airtable file helper, not a full invoice-sync workflow. It exists to enrich a previously created FortnoxSupplierInvoice header with an invoice PDF field update.

Most important clarified conclusion: the current sources prove local binary download, local file write, and Airtable update by attachment-style URL payload. They do not yet prove the exact webhook/backend mechanism by which Airtable ultimately receives the actual file bytes.
