3. Observed Execution Path

The runtime path visible in the JSON is very short and fully linear. There are no IF branches inside the workflow; the only conditional behavior is embedded inside one expression in the Set node.

A workflow fails
  -> Error Trigger fires
  -> wait 5 minutes
  -> call Credentials (workflow 109)
  -> list 1 record from Airtable Settings table
  -> compute two outbound fields:
       - Temp Field = current timestamp as string
       - Download Invoices = if Idle == 1 then true else keep current Download Invoices
  -> update the same Settings record
  -> stop

The flow calls the shared Credentials workflow 109 so it can resolve the Airtable base and Settings table ids dynamically.

The Get Settings node lists one record from the shared Settings table. No filter formula is visible in this export.

The Update Temp Field node keeps only two outgoing fields: a timestamp string in Temp Field and a conditionally recomputed Download Invoices boolean.
6
Update the same Settings record
The workflow writes those fields back to the Settings record id returned by Get Settings, then ends.

4. Exact Settings Write Behavior

The entire business effect of this workflow is concentrated in the last two nodes. That makes their exact behavior unusually important.

Observed elementWhat it doesWhy it matters
Get Settings Reads exactly one Airtable record from the shared Settings table. The recovery action is global / orchestrator-level, not targeted to a specific customer, order, invoice, or supplier record.
Temp Field Sets Temp Field = $now.valueOf().toString(). This guarantees a changed value on every run, so the Settings record definitely gets modified.
Download Invoices Sets true only when Idle === 1; otherwise it preserves the existing Download Invoices value. The workflow can explicitly re-enable the invoice-stage downloader entrypoint, but only under one visible condition.
Update Settings Updates the same Settings record returned by the read step. This is the only durable write performed by the workflow.

Important design fact

The workflow does not read the error message, error type, failed workflow name, failed entity id, or any other failure-specific detail. It applies the same generic recovery write regardless of what actually broke upstream.

Likely reason for the design

This looks intentional. The workflow appears designed to kick the orchestrator back into motion rather than to diagnose or repair the original failure. In other words, it behaves more like a controlled restart helper than a forensic error-management workflow.

5. Recovery and Restart Model

This workflow is most useful when read together with the Downloader note. The two sources together suggest a simple recovery pattern: wait, touch the Settings record, and let the normal Airtable-triggered orchestration resume.

Most plausible runtime interpretation: after a failure, this workflow waits a few minutes, touches the shared Settings record, and thereby creates the conditions for the normal downloader/state-machine flow to restart from its usual control surface.

What is definitely proven

The workflow waits, loads settings, computes a timestamp, conditionally forces Download Invoices = true when Idle == 1, and updates the Settings record. Those steps are explicit in the uploaded JSON.

What remains interpretation

The uploaded files do not contain runtime logs proving the exact downstream sequence after the Settings update. The restart interpretation is strong because the Downloader note confirms that the Settings record is the orchestrator trigger surface, but the actual next execution is still inferred rather than directly logged in the available sources.

Important boundary: this workflow does not itself retry the failed HTTP request, does not inspect which record failed, and does not repair partial writes in domain tables. It performs only a generic orchestrator-state update.

7. Confirmed Quirks and Unresolved Questions

7.1 The export is inactive, but the Downloader still points to it

The uploaded Error Handler export itself is inactive, yet the Downloader export and its analysis both still identify 117 as the configured error workflow. So this flow remains part of the documented dependency graph even though its captured standalone export is inactive.

7.2 The workflow ignores the actual error payload

No node uses the incoming error contents. That means all failure types are funneled into the same generic response.

7.3 The workflow updates only orchestrator fields, not diagnostic fields

The visible writeback fields are Temp Field and Download Invoices. There is no visible write of error text, failed workflow id, stack trace, retry counter, or alert flag.

7.4 The workflow is self-referential in its own export settings

The uploaded JSON shows errorWorkflow = "117" inside workflow 117 itself. The available sources do not explain whether this is intentional, ignored by n8n for error workflows, or potentially risky if the handler itself throws an error.

Still unresolved: the current source set does not prove what the Idle field means operationally, where it is maintained, or whether a value of 1 always means the downloader is safe to restart from the invoice stage.

7.5 Why the five-minute wait probably exists

The most plausible explanation is debouncing / cool-off: let transient outages, API throttling, or half-finished writes settle before touching the orchestrator state again. That rationale fits the structure, but it is not explicitly documented in the workflow.

7.6 Why the workflow always writes a fresh timestamp

The likely purpose of Temp Field is to guarantee a Settings-record modification even when Download Invoices does not change. That would make sense in a system where the main orchestrator watches Last Modified Time on that record.

8. Plain-English Verdict

This workflow is the integration's generic “after failure, poke the control record” recovery flow.

  • It starts from n8n's global error trigger, not from a Fortnox entity event.
  • It waits five minutes before doing anything.
  • It loads the shared Settings record, updates a timestamp field, and conditionally turns Download Invoices back on.
  • It does not inspect the error payload and does not repair any specific customer, order, invoice, or supplier record.
  • Its most likely purpose is to restart or reawaken the normal downloader/state-machine flow through the Settings table.
Bottom line: the uploaded export proves that workflow 117 is a very small control-plane recovery workflow, not a data-sync workflow. Its job is to wait after a failure and then write back to the shared Airtable Settings record so the main orchestrator can likely resume. The biggest remaining unknowns are the exact semantics of Idle and whether the self-referential errorWorkflow = 117 setting is harmless or an overlooked configuration quirk.
