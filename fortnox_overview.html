Fortnox ↔ Airtable — Helicopter View

3-minute overview  |  automation.offertizer.com  |  base app01yDjf98U6ejKs

One sentence: n8n on automation.offertizer.com runs on a schedule, polling both Fortnox API and Airtable, keeping them in sync. No webhooks, no real-time — pure polling.

The Two Directions

Fortnox → Airtable  All 8 tables confirmed Airtable → Fortnox  Schema only — not active
Customers, Suppliers, Supplier Invoices, Orders, Order Rows, Invoices, Invoice Rows, Articles.

Fortnox is the source of truth for all entities. n8n fetches from the Fortnox REST API and upserts into Airtable. Staff can view and annotate, but Airtable changes to these records are overwritten on the next sync run.		 A push workflow (Airtable → Fortnox) was designed for Orders: Order Status and Sync Fortnox fields exist in the FortnoxOrder schema with the full planned lifecycle. This workflow is not active. All FortnoxOrder records have DocumentNumber pre-populated (assigned by Fortnox), confirming they are pulled from Fortnox, not created in Airtable. See the Orders & Invoices document for details.

Order Lifecycle — Fortnox → Airtable

⚠ Schema vs. reality: The FortnoxOrder table has FIELD Order Status with choices "Airtable Draft", "Copy to Fortnox", "Fortnox Order", etc., and FIELD Sync Fortnox with "To sync" / "In progress" — implying a push workflow where staff initiate orders in Airtable. This workflow is not active. Three proofs from live data confirm this: (1) every FortnoxOrder record has DocumentNumber pre-populated by Fortnox — records created in Airtable first would lack it until Fortnox assigned one; (2) FIELD Order Status and FIELD Sync Fortnox are empty on all records — no lifecycle stages have ever been written; (3) every Fortnox OrderRow record has LineItem Source = "Fortnox" — the "Airtable users" value intended for staff-created rows is unused across the entire table. To be 100% certain, base users should be asked whether they have ever initiated an order from Airtable.
1
Staff creates the order directly in the Fortnox web app. Fortnox assigns a DocumentNumber immediately.

How Sync Is Triggered

  • Schedule-based polling — no webhooks, no real-time events from either side.
  • n8n connects to Airtable as a service account (integration.adgrabber@gmail.com) and reads per-table sync views to determine which records to process.
  • Each table has a named sync view (e.g. FortnoxOrder_Sync, FortnoxArticle_Sync). The view's filter defines the polling window (past week, past month, or unfiltered for small tables).
  • The sync view on Fortnox OrderRow has no filter — it exposes all 17,000+ rows for n8n's reconciliation and LMT checks.

What Keeps It From Looping

When n8n writes Fortnox data back into Airtable, those writes bump a scoped Last Modified Time field (Fortnox details modified date) that watches only the ~39 Fortnox payload fields. On the next poll, n8n knows those are its own writes and does not re-process them. User-side changes (linking allocations, soft-deleting rows) bump different scoped LMT fields — n8n handles those separately or ignores them for Fortnox sync.

Each Fortnox OrderRow has 6 scoped LMT fields, each watching a different column subset:

  • Last Modified Time — 44 Fortnox data fields (the main sync change detector)
  • Fortnox details modified date — ~39 pure Fortnox payload fields (what n8n itself writes)
  • Last Modified — Total field only (financial change signal)
  • Last Modified 2 — Line Item Allocation link only (internal Airtable, never pushed to Fortnox)
  • Last Modified 3 / When deleted — Deleted checkbox only (soft-deletion detection)
  • Last Modified 4 — all fields (audit/monitoring only)
Bottom line: Fortnox is the financial system of record for all entities. The integration is one-directional — Fortnox → Airtable — for all 8 tables confirmed from live data. An Airtable→Fortnox push workflow was planned (schema fields exist on FortnoxOrder) but is not active.
