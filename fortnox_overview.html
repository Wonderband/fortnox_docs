Fortnox ↔ Airtable — Helicopter View

3-minute overview  |  automation.offertizer.com  |  base app01yDjf98U6ejKs

One sentence: n8n on automation.offertizer.com runs on a schedule, polling both Fortnox API and Airtable, keeping them in sync. No webhooks, no real-time — pure polling.

The Two Directions

Fortnox → Airtable  6 of 8 tables Airtable → Fortnox  Orders only
Customers, Suppliers, Supplier Invoices, Invoices, Invoice Rows, Articles.

Fortnox is the source of truth. n8n fetches from the Fortnox REST API and upserts into Airtable. Staff can view and annotate, but Airtable changes to these records are overwritten on the next sync run.		 Only Orders and Order Rows flow both ways. Orders are created in Airtable first, then pushed to Fortnox. Once Fortnox assigns a DocumentNumber, that ID is written back into Airtable.

Order Lifecycle — The Only Two-Way Flow

1
Staff creates a FortnoxOrder record in Airtable. Order Status = "Airtable Draft". ID is auto-generated (Autonumber-CustomerName) — no Fortnox number yet.
2
Staff adds line items (Fortnox OrderRow records), sets LineItem Source = "Airtable users" and links a FortnoxArticle to each row.
3
Staff sets Order Status = "Copy to Fortnox" and Sync Fortnox = "To sync". This is the trigger signal.
4
n8n detects the flag → sets it to "In progress" → verifies customer number → POSTs the order + rows to https://api.fortnox.se/3/orders.
5
Fortnox returns a DocumentNumber and RowId values. n8n writes these back to Airtable. Sync Fortnox is cleared. Order Status → "Fortnox Order". The ID field now shows the Fortnox DocumentNumber.
6
Order progresses in Fortnox through invoicing. n8n syncs the resulting FortnoxInvoice back into Airtable, linked to the originating order.

How Sync Is Triggered

  • Schedule-based polling — no webhooks, no real-time events from either side.
  • n8n connects to Airtable as a service account (integration.adgrabber@gmail.com) which owns a personal view called sync view on Fortnox OrderRow — no filter, all 17,000+ rows. This is what n8n reads on every run.
  • For the Airtable → Fortnox direction, n8n queries FortnoxOrder for records where Sync Fortnox = "To sync".

What Keeps It From Looping

When n8n writes Fortnox data back into Airtable, those writes bump a scoped Last Modified Time field (Fortnox details modified date) that watches only the ~39 Fortnox payload fields. On the next poll, n8n knows those are its own writes and does not re-process them. User-side changes (linking allocations, soft-deleting rows) bump different scoped LMT fields — n8n handles those separately or ignores them for Fortnox sync.

Each Fortnox OrderRow has 6 scoped LMT fields, each watching a different column subset:

  • Last Modified Time — 44 Fortnox data fields (the main sync change detector)
  • Fortnox details modified date — ~39 pure Fortnox payload fields (what n8n itself writes)
  • Last Modified — Total field only (financial change signal)
  • Last Modified 2 — Line Item Allocation link only (internal Airtable, never pushed to Fortnox)
  • Last Modified 3 / When deleted — Deleted checkbox only (soft-deletion detection)
  • Last Modified 4 — all fields (audit/monitoring only)
Bottom line: Fortnox is the financial system of record for everything except new orders, which originate in Airtable. The integration is one-directional for most entities and bidirectional only for the sales order flow.
