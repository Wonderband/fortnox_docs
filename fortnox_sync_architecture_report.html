4.2 FortnoxSupplierInvoice — Sync Field Is a Lookup

The Sync field (fldTSfERJEsU4O6rn) on FortnoxSupplierInvoice is type multipleLookupValues. It pulls Internal Status from the linked FortnoxSupplier record. It is not an independent field — it reflects whatever sync gate value the parent supplier has.

The table additionally has Supplier Internal Status (fldi8XVUB7Ik48CDi) and Supplier Internal Status String (fldTeeLyx67EtZpeo) — also lookups from the parent supplier.

4.3 FortnoxOrder_Sync View — Confirmed Filter

Inspected via Playwright on 2026-03-17. The view filter is "Created is within the past month" (time window on the record creation date). It is not filtered by Sync Fortnox field value. The view is a personal view (not collaborative — visible but not configurable by other users). It showed 0 records at time of inspection.

6. Inferred Architecture

Everything in this section is inferred. The actual n8n workflows on automation.offertizer.com were not inspected. The conclusions below are the most plausible interpretations of the confirmed field evidence but are not proven.

6.1 What Mechanism Is Currently Active

Only integration-adgrabber has active OAuth tokens (it is the only app in "My Integrations"). The adgrabber-websocket2 and adgrabber-websocket3 apps exist in the Developer Portal but have never been authorized — they have no tokens and cannot make API calls or open WebSocket connections. It is therefore most likely that all current sync runs through integration-adgrabber using the REST API, triggered on a schedule (cron). The exact interval is not known.

6.2 Fortnox → Airtable Direction

The presence of @url, Last Modified Time, and RowId fields on all 8 tables is consistent with n8n periodically fetching data from Fortnox REST API endpoints and upserting Airtable records. The Fortnox details modified date LMT field on Fortnox OrderRow (which watches only Fortnox data fields) could be used to detect whether a row's Fortnox data has changed since last sync. Whether ?lastmodified query params are used in the GET calls is not known.

6.3 Airtable → Fortnox Direction

FortnoxOrder is the only table with a dedicated write-queue flag: Sync Fortnox (options: "To sync" / "In progress"). This pattern — set to "To sync" to request a push, mark "In progress" while processing — is a common n8n pattern for preventing duplicate runs. FortnoxOrder also has the most complex LMT field setup (multiple scoped fields) which makes sense if n8n writes back to it and needs to avoid re-triggering its own reads.

The LineItem Source field ("Fortnox" / "Airtable users") on Fortnox OrderRow likely controls which rows are pushed to Fortnox: only "Airtable users" rows would be written, while "Fortnox" rows would be treated as read-only.

The FortnoxOrder_Sync view is confirmed to filter by "Created is within the past month" (not by Sync Fortnox value). Whether n8n applies an additional Sync Fortnox = "To sync" filter within the workflow node — or processes all recent orders unconditionally — is not known.

6.4 The Deleted/Changed in Fortnox Cluster

Fortnox OrderRow has a cluster of fields for tracking Fortnox-side deletion/change events: Deleted/Changed in fortnox (singleSelect: "Deleted" / "Changed"), When is it changed/deleted in fortnox (manual date), is it changed/deleted today? (formula comparing that date to today), and the CHANGED/DELETED TODAY IN FORTNOX view. Whether these flags are set automatically by n8n or entered manually is not confirmed.

6.5 WebSocket Potential

The adgrabber-websocket2 app (9 read-only scopes) appears designed for WebSocket use. Activating it would require only the one-time OAuth authorization step in Fortnox. If deployed, it would replace scheduled polling for the Fortnox → Airtable direction with event-driven updates, reducing API call volume significantly given the 4 req/sec limit is shared across all operations.
