Supplier Invoice Flows — Two Sources into FortnoxSupplierInvoice

Reference document  |  automation.offertizer.com  |  base app01yDjf98U6ejKs

One sentence: TABLE FortnoxSupplierInvoice (tblYERBn2P6wrHEzl) receives records from two completely separate billing systems — standard supplier invoices synced automatically from Fortnox, and media platform billing imported manually from export files — but both flow downstream into sys.fakturamapping for identical cost attribution.

Why Two Flows Exist

The agency handles two fundamentally different categories of supplier costs:

  • Standard suppliers (printing, production, freelancers, etc.) invoice through the normal B2B process: they send invoices to the agency, which registers them in Fortnox — the accounting system of record. These flow into Airtable automatically via n8n sync.
  • Media buying platforms (Google Ads, Meta, DV360, etc.) don't send traditional invoices — they bill the agency directly through their own platform billing systems and generate periodic billing exports (PDFs or CSVs). These exports never enter Fortnox; instead, staff import them manually into Airtable.

Despite their different origins, both types represent real supplier costs that must be attributed to specific campaigns for P&L reporting. Keeping them in one table (FortnoxSupplierInvoice) means the downstream cost-attribution workflow (sys.fakturamapping) handles both with the same logic.

The Two Flows at a Glance

Flow 1  Fortnox → n8n → Airtable

Automated. Supplier invoices processed through accounting.

1
Supplier sends invoice; Fortnox staff registers it in the Fortnox app.
2
n8n checks FortnoxSupplier: Internal Status = "Sync" must be set. Suppliers marked "Do not sync" are skipped entirely.
3
n8n polls GET /3/supplierinvoices and upserts matching invoices into TABLE FortnoxSupplierInvoice.
4
Record ID = Fortnox GivenNumber (document number). Currency conversion applied if invoice is not in SEK.

Flow 2  Platform Export → Manual Import

Manual. Platform billing that bypasses Fortnox entirely.

1
Media platform (Google, Meta, etc.) generates a billing export for the period.
2
Staff creates a record in TABLE Platform Import Files (tbl922EeY899adWVD) with platform name, date range, total, and description. In practice no file is attached — the Pdf Url and Attachment fields are empty on all 145 live records.
3
Staff manually creates one FortnoxSupplierInvoice record per client that ran campaigns on that platform during the period, splitting the platform's billing total across clients. Each invoice links back to the parent Platform Import File record. The Importing Variance formula on the import file verifies the split sums to zero.
4
No currency conversion — platform billing is always in SEK. No FortnoxSupplier link — platform is identified via FIELD Platform Code (links to platform names table tblSFdN7U9Bri8zwr).

TABLE Platform Import Files

TABLE Platform Import Files (tbl922EeY899adWVD) acts as the entry point and audit trail for all platform billing. It holds one record per platform per billing period (145 records in live data: Facebook, LinkedIn, Microsoft, Snapchat, TikTok, and others).

  • Each record covers one platform's billing for one period. Key fields: Platform Name (link to platform names table), Rapport Start / Rapport End (period dates), Total (platform's billed amount in SEK), Description (free text, e.g. "Facebook Oktober 2025"), Import Status (singleSelect — "Received" → "Working on" → "Finished mapping"), Import Assignee.
  • Pdf Url and Attachtment if not Url fields exist but are empty on every live record — no actual files are stored in Airtable.
  • The platform billing total is split into multiple FortnoxSupplierInvoice records — one per client that ran campaigns on that platform during the period. Facebook, which typically has more active clients, generates 8–26 invoice records per monthly import; Snapchat typically 1–2.
  • Each invoice links back to the parent import via FIELD Platform Import File (fldAzxkHMaZWQYlt9). The Importing Variance rollup on the import file confirms the split is complete (import total − sum of invoices = 0).
Validation chain built into the table: Import remarks checks that the import total matches the sum of linked invoices. Mapping remarks then checks that campaign costs equal allocated spend. Both must show ✅ for the import to be considered fully complete.
The Platform Import File link field is only populated on platform records. Fortnox-synced invoices leave it empty — they come from the Fortnox API, not from an import batch.

How to Tell the Two Types Apart

Platform import records have no FortnoxSupplier link — they were created manually and were never part of a Fortnox sync. Because Suppliertype and Internal Status are lookups that derive from FortnoxSupplier, both fields are also empty on platform records. The actual distinguishing signals are structural, not a single flag field:

Field / signal Fortnox-Synced record Platform import record
FortnoxSupplier link Populated — linked to the supplier in TABLE FortnoxSupplier Empty — no supplier link at all
Platform Code Empty Populated — links to TABLE Platform Names (tblSFdN7U9Bri8zwr), identifying the platform (Facebook, LinkedIn, etc.)
Platform Import File link Empty Populated — links back to the source TABLE Platform Import Files record
Suppliertype (lookup via FortnoxSupplier) Standard value or blank Empty — no FortnoxSupplier to look up from
Internal Status / Sync (lookup via FortnoxSupplier) "Sync" or "Do not sync" Empty — no FortnoxSupplier to look up from
Kund (client) link Empty — Fortnox supplier invoices carry no client context. Campaign cost attribution is done entirely downstream via sys.fakturamapping. Always populated — one invoice record is created per client that ran campaigns on the platform that month. The client split is the mechanism that makes per-campaign attribution possible.
Suppliertype = "Platform - Not Fortnox" is a property that lives on TABLE FortnoxSupplier — it classifies certain supplier records as platform entities. It does not appear on the invoice records created via the manual import flow, because those records have no FortnoxSupplier link to look up from. To identify a platform import invoice: Platform Code is populated AND Platform Import File is populated AND FortnoxSupplier is empty.

Downstream Treatment — What Is the Same

Identical downstream: Both record types flow into TABLE sys.fakturamapping through the same FIELD sys.fakturamapping link (fld2h78ru7NwwupNX). Staff attribute invoice costs to specific campaigns using fakturamapping regardless of whether the invoice came from Fortnox or a platform billing import. The P&L machinery treats them the same.

The following features apply equally to both invoice types:

  • Invoice fully mapped? — formula checking whether total attributed cost equals invoice total. Works the same for both.
  • Attest Status — singleSelect field ("Ok att betala", "Ska inte betalas", "För åtgärd") used for payment approval. Applies to both types.
  • Cost attribution to campaigns — via sys.fakturamapping, both types split costs across campaigns using the same monthly allocation logic.
Bottom line: The split exists because two different billing realities (B2B accounting via Fortnox vs. platform self-billing) need to be unified in one place for P&L. Platform import records are identified structurally — Platform Code and Platform Import File populated, FortnoxSupplier empty — not by a single flag field. All downstream cost attribution via sys.fakturamapping is identical for both types.
