2. What the Flow Actually Does

Schedule Trigger (every 50 min) or manual Start
  -> HTTP Fortnox Test
     -> GET https://api.fortnox.se/3/invoices
        through ratelimit.offertizer.com
        using n8n OAuth2 credential: Fortnox Oauth2

Directly proven from the workflow

  • It is a tiny REST heartbeat, not a business workflow.
  • It uses the same shared Fortnox OAuth credential as the main Fortnox flows.
  • It does not write to Airtable, does not run components, and does not manage the WebSocket workflow.
  • The Google Drive node is disconnected and does nothing.

3. What It Means

This workflow is best described as a simple Fortnox auth touchpoint. It does not perform an explicit token-refresh procedure itself. It only creates one scheduled Fortnox API request, which allows n8n to use the shared OAuth credential and refresh it if needed.

Important: this flow is not the WebSocket keep-alive. The WebSocket listener is maintained separately by the WebSocket/Downloader Reload workflow, which restarts the handler regularly. This refresh flow does not open, ping, or sustain the Fortnox socket session.

4. What Happens If You Remove It

  • Core Fortnox sync should still work as long as business workflows continue to use the same shared OAuth credential.
  • Business flows can still refresh auth on demand when they call Fortnox.
  • No direct WebSocket protection is lost, because this flow is not the socket watchdog.
  • The only thing you lose is one guaranteed low-impact Fortnox heartbeat every 50 minutes.

5. Do We Actually Need It?

Direct answer: no, it is not required for the Fortnox integration to function.
QuestionAnswer
Is it required for token refresh?No. Any real Fortnox workflow using the same credential can trigger the same OAuth refresh behavior.
Is it required for business syncs to succeed?No. The main flows already authenticate against Fortnox directly.
Is it required to keep the WebSocket alive?No. That role belongs to the reload/reconnect workflow, not this one.
Does it provide any value?Yes, but limited. It gives one predictable Fortnox auth heartbeat with zero business side effects.

6. Keep or Delete?

Verdict: disable first, then delete if nothing regresses.

The strongest recommendation is operationally conservative:

  • Disable the workflow.
  • Observe whether Downloader, WebSocket Handler, and component flows continue working normally.
  • If there is no regression, remove it as redundant.
When to keep it temporarily: only if the team explicitly wants a separate Fortnox auth smoke test / heartbeat for operational comfort. That is an ops preference, not an architectural requirement.

7. Final Plain-English Conclusion

This workflow is not a core dependency. It does not contain unique auth logic, it does not protect the WebSocket connection, and it is not required for the real business flows to refresh Fortnox access during normal work.

Recommendation: treat it as optional legacy utility. Unless the team wants to keep a dedicated heartbeat on purpose, it should be disabled and then removed.
