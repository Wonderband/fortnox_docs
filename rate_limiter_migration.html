Fortnox Rate Limiter — Reference Guide

How it works on Offertizer today · How to replicate it on any server · Full migration checklist · Created 2026-04-20

1 — Why It Exists

Why a Rate Limiter Is Needed

Fortnox enforces a hard limit of 4 API requests per second per account. If you exceed it, Fortnox returns HTTP 429 Too Many Requests and the flow fails.

Your automation easily hits this limit because multiple flows run at the same time, and each entity can trigger several sub-calls:

Example: one invoice sync triggers 4+ API calls
Downloader fires → Invoice Component → calls Customer Component → calls Order Component → calls Article Component (×N rows)
Meanwhile: WebSocket Handler also receives events and fires its own calls to the same Fortnox account.

Without a shared throttle, those concurrent calls stack up and exceed the 4/sec limit within seconds during any bulk sync.

2 — Current Setup

How It Works on Offertizer Today

Offertizer provides a reverse proxy at ratelimit.offertizer.com. Every Fortnox API call in every flow is routed through it instead of calling Fortnox directly.

The URL pattern used in every flow

N8N Flow
any component or downloader
→
ratelimit.offertizer.com
/{ratelimit_id}?url={fortnox_url}
→ queued →
api.fortnox.se
max 4 req/sec
actual URL in every HTTP node
https://ratelimit.offertizer.com/fb7db917-6e2f-4743-bec1-682161b56d24
  ?url=https://api.fortnox.se/3/articles/1234
  Authorization: Bearer {fortnox_oauth_token}

The ratelimit_id UUID (fb7db917...) is your account identifier on the proxy — it tells the proxy which queue and rate-limit budget to use. It comes from the Credentials flow #109 so every flow inherits it automatically.

What the proxy does step by step

  1. Receives the request from N8N (with the real Fortnox URL in ?url=)
  2. Puts it into a shared queue for your account
  3. Sends requests out of the queue one at a time, with a minimum gap of ~250ms between them
  4. Forwards the Fortnox OAuth token along with the request to Fortnox
  5. Returns the Fortnox response back to N8N — N8N sees a normal HTTP response
Key point
The queue is shared across all flows. So when Downloader and WebSocket Handler both fire calls at the same moment, they both enter the same queue and exit at a controlled rate — Fortnox never sees more than 4 per second regardless of how many flows run concurrently.

3 — Implementation

How to Implement the Rate Limiter on Any Server

Recommended approach
Run a small Node.js proxy service on the same server as N8N. All flows call http://localhost:3333 instead of ratelimit.offertizer.com. One shared queue — identical behaviour to today.

Step 1 — Create the proxy file

Create this file on your server, in any folder (e.g. /opt/fortnox-proxy/proxy.js):

proxy.js
const express    = require('express');
const axios       = require('axios');
const Bottleneck  = require('bottleneck');

const app = express();
app.use(express.json());

// Fortnox hard limit = 4 req/sec.
// We use 3.5/sec (one per 285ms) as a safety margin.
const limiter = new Bottleneck({
  maxConcurrent: 1,    // one Fortnox request at a time
  minTime:       285   // minimum 285ms between requests (~3.5/sec)
});

// Accept any path — the /:ratelimitId segment is ignored,
// it's just there so the URL format from N8N still works.
app.use('*', async (req, res) => {

  const targetUrl = req.query.url;

  if (!targetUrl || !targetUrl.startsWith('https://api.fortnox.se')) {
    return res.status(400).json({ error: 'Missing or invalid ?url= parameter' });
  }

  // Forward all query params except 'url' itself (e.g. page=2)
  const params = { ...req.query };
  delete params.url;

  // Forward all headers except 'host'
  const headers = { ...req.headers };
  delete headers['host'];
  delete headers['content-length'];

  try {
    const response = await limiter.schedule(() => axios({
      method:         req.method,
      url:            targetUrl,
      headers:        headers,
      params:         params,
      data:           ['POST','PUT','PATCH'].includes(req.method) ? req.body : undefined,
      validateStatus: () => true  // pass Fortnox errors through unchanged
    }));

    res.status(response.status).json(response.data);

  } catch (err) {
    res.status(502).json({ error: 'Proxy error', detail: err.message });
  }
});

const PORT = process.env.PROXY_PORT || 3333;
app.listen(PORT, '127.0.0.1', () => {
  console.log(`Fortnox rate-limit proxy running on localhost:${PORT}`);
  console.log(`Rate: 1 request per 285ms (max ~3.5/sec)`);
});

Step 2 — Install dependencies and start

terminal — run once
# Go to your proxy folder
mkdir -p /opt/fortnox-proxy && cd /opt/fortnox-proxy
cp proxy.js .

# Install packages
npm init -y
npm install express axios bottleneck

# Install PM2 — keeps the proxy alive after server restarts
npm install -g pm2

# Start the proxy
pm2 start proxy.js --name fortnox-proxy

# Save process list and enable auto-start on reboot
pm2 save
pm2 startup   # follow the instruction it prints
Verify it's running
pm2 status — should show fortnox-proxy as online
curl "http://localhost:3333/test?url=https://api.fortnox.se/3/invoices" -H "Authorization: Bearer YOUR_TOKEN" — should return a Fortnox response

Step 3 — Update the flow URLs (before importing JSON files)

Run this once on your exported JSON files before importing them into the new N8N. It replaces the Offertizer proxy domain with your local proxy in all flows at once:

terminal — run in your json_flows folder
# On Linux / Mac
sed -i 's|https://ratelimit.offertizer.com|http://localhost:3333|g' *.json

# On Windows (PowerShell)
Get-ChildItem *.json | ForEach-Object {
  (Get-Content $_.FullName) -replace 'https://ratelimit\.offertizer\.com', 'http://localhost:3333' |
  Set-Content $_.FullName
}

After this, every HTTP node that was calling ratelimit.offertizer.com will call localhost:3333 instead. The ratelimit_id UUID in the Credentials flow can stay exactly the same — your proxy accepts any path and ignores it.

4 — Other URLs to Update

Other Offertizer References to Fix After Migration

Beyond the rate limiter, two more flows reference Offertizer's server address directly. These need manual changes inside N8N after importing.

Watchdog flow — N8N API URL

The Watchdog checks whether the Downloader is stale by calling the N8N REST API:

current (Offertizer)
GET https://automation.offertizer.com/api/v1/executions?workflowId=118&limit=1
change to (your new server)
GET http://localhost:5678/api/v1/executions?workflowId=118&limit=1
X-N8N-API-KEY: your-n8n-api-key
You also need an N8N API key
In your new N8N: go to Settings → API → Create API Key. Add it as a header X-N8N-API-KEY in the Watchdog's HTTP node. Without it, the call will be rejected with 401.

SI File Component (#120) — PDF webhook URL

After downloading a PDF, this flow gives Airtable a URL to fetch the file from. Currently it points to a webhook on Offertizer's server:

current (Offertizer)
https://automation.offertizer.com/webhook/daa58b95-f6e5-42ea-b5c9-0efe21eb590d?file=invoice.pdf&t=...
change to (your new server)
https://your-domain-or-ip:5678/webhook/NEW-WEBHOOK-UUID?file=invoice.pdf&t=...
How to get your new webhook UUID
When you import the SI File Component flow into your new N8N, the Webhook node inside it will be assigned a new UUID automatically. Copy that UUID from the node and paste it into the Code node that builds the Airtable attachment URL. Also — your new server must be reachable from the internet so Airtable can fetch the PDF.
Ongoing race condition risk (exists today too)
The PDF is saved to a hardcoded path: /home/ubuntu/files/invoice.pdf. If two supplier invoices are processed close together, the second file overwrites the first before Airtable has fetched it. This is a pre-existing issue — not introduced by migration — but worth fixing eventually by using a unique filename per invoice (e.g. /home/ubuntu/files/{GivenNumber}.pdf).
5 — Migration Checklist

Step-by-Step Migration Checklist

  • Before you touch the new server
  • Run the sed / PowerShell find-and-replace on all JSON flow files to swap ratelimit.offertizer.com → http://localhost:3333
  • Note which flows are currently ACTIVE on Offertizer — you will need to re-activate them after import
  • Export all 15 flow JSON files from Offertizer N8N (if not already done)
  • Set up the new server
  • Install N8N on the new server (Docker recommended: docker run -d --name n8n -p 5678:5678 n8nio/n8n)
  • Create the proxy folder: mkdir -p /opt/fortnox-proxy
  • Copy proxy.js to /opt/fortnox-proxy/
  • Run npm install express axios bottleneck in that folder
  • Install PM2: npm install -g pm2
  • Start proxy: pm2 start proxy.js --name fortnox-proxy && pm2 save && pm2 startup
  • Verify proxy is running: pm2 status
  • Import flows into new N8N
  • Import all 15 modified JSON files into the new N8N instance
  • Re-enter Fortnox OAuth2 credentials (production) — they do not export with the JSON
  • Re-enter Fortnox Sandbox OAuth2 credentials (for Uploader #119)
  • Re-enter Airtable token credential
  • Re-enter Google Drive credential (used by Refresh Credentials #107)
  • Fix the three manual URL changes in N8N
  • Watchdog flow: change automation.offertizer.com/api/v1/executions → localhost:5678/api/v1/executions
  • Watchdog flow: generate N8N API key (Settings → API) and add as X-N8N-API-KEY header
  • Watchdog flow: confirm workflow IDs for Downloader and WebSocket Handler — they may change on re-import
  • SI File Component: copy new webhook UUID from the Webhook node and update the Code node URL
  • Ensure new server is accessible from the internet on port 5678 (or whichever port N8N runs on) so Airtable can reach the PDF webhook
  • Activate and verify
  • Activate: Downloader #118, WebSocket Handler #116, Watchdog, Refresh Credentials #107
  • Trigger one manual run of Article Component — check it calls localhost:3333 and gets a Fortnox response
  • Check proxy logs: pm2 logs fortnox-proxy — you should see requests being queued and forwarded
  • Run a small bulk sync (set Download Articles = true in Settings) — monitor that no 429 errors appear
  • Trigger a WebSocket event by making a change in Fortnox — confirm it syncs to Airtable
  • Test a supplier invoice that has a PDF — confirm the file appears in Airtable
  • Deactivate all flows on the old Offertizer server

No cloud service needed
AWS, Cloudflare, and similar services throttle inbound traffic to your APIs. They do not help throttle outbound calls from your server to Fortnox. The self-hosted proxy above is the right tool for this job — it is the same pattern that Offertizer used, running directly on your new server.
