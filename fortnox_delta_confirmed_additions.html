2. Confirmed Additions

2.1 Fortnox sync does not run through Airtable automations

The later data-flow report adds a useful explicit conclusion: there are no Fortnox-related Airtable automations in this base. The Fortnox sync path is handled by n8n on automation.offertizer.com, while Airtable automations cover only internal business logic such as campaign creation, button flows, alerts, or internal housekeeping.

This is worth preserving because earlier architecture files described the integration stack but did not clearly resolve whether any Fortnox movement was triggered by Airtable automations.

2.2 The synced Fortnox tables use a dual-field relationship pattern

A strong implementation detail from the later reports is that parent/child relations across the 8 Fortnox sync tables follow a consistent structure:

  • one field stores the Fortnox-side identifier as plain text or number, for example CustomerNumber, SupplierNumber, DocumentNumber, OrderReference, or ArticleNumber
  • a second field stores the Airtable linked-record pointer used for rollups, UI display, and downstream formulas

This pattern is especially important on child tables like FortnoxInvoice, FortnoxInvoiceRow, and Fortnox OrderRow, where both a Fortnox key and an Airtable link coexist.

2.3 Parent tables must exist before child links can be written

The linked-records report contributes a clear sync dependency rule: n8n must have already created the parent Airtable record before it can write a child record that links to it. This is particularly relevant for:

  • FortnoxOrder before Fortnox OrderRow
  • FortnoxInvoice before FortnoxInvoiceRow
  • FortnoxCustomer before linking orders and invoices to a customer
  • FortnoxSupplier before linking supplier invoices
  • FortnoxArticle before linking order rows or invoice rows to articles

This does not change the business model, but it adds useful knowledge about why sync ordering matters operationally.

2.4 Order–invoice linking is clarified more precisely

The business-logic report adds one high-value clarification on the customer side:

  • FortnoxInvoice links back to FortnoxOrder using the linked-record field plus OrderReference as the Fortnox-side text copy of the parent order number
  • InvoiceReference is not the order link; it is the credit-note reference field and normally shows a neutral value on standard invoices

This distinction is worth preserving because the earlier KB already established that invoices are connected back to orders, but the new report removes a common field-name ambiguity.

2.5 The changed/deleted-in-Fortnox process is manual

The later data-flow report makes a valuable operational point explicit: the field When is it changed/deleted in fortnox is a plain date field that requires a human to fill it. Therefore the CHANGED/DELETED TODAY IN FORTNOX view is not a live detector or automated sync trigger. It is a human-operated monitoring process.

The practical implication is that the system does not have automated deletion/change detection for older Fortnox-side changes; someone must notice the change and mark it manually for review.

2.6 Scoped LMT fields are active precision sync signals

The earlier KB already documented that Fortnox OrderRow contains several separate Last Modified Time fields with different scopes. The new value added by the later report is the operational interpretation:

  • one scoped LMT watches the Fortnox payload fields
  • another watches only allocation-link changes
  • another watches soft-delete changes
  • one all-fields LMT remains available for broad auditing

That means the multi-LMT design is not cosmetic. It is there so n8n can tell the difference between:

  • a Fortnox-derived value change written by the integration itself
  • a user linking or unlinking allocation records in Airtable
  • a soft-delete action
  • a generic human edit that should not be interpreted as Fortnox-side business data

2.7 n8n writes operational status back into Airtable

The later report adds a concrete implementation detail: the sync writes operational trace/status messages back into Airtable fields such as Automation Log and Fortnox Download Status on the order side. This is useful because it shows that some Airtable columns are not business data mirrors but integration-observability fields.

2.8 Platform imports use a three-level reconciliation chain

The business-logic report adds a compact but important finance-control model for the platform-import flow. The reconciliation chain works across three levels:

  1. Platform Import Files total from the external platform billing PDF
  2. FortnoxSupplierInvoice totals linked to that platform-import record
  3. sys.fakturamapping campaign allocations, compared back to campaign-side revenue allocation through sys.orderallocationmapping

This is useful new KB material because it clarifies that platform-import quality control is not only “invoice total equals PDF total,” but a deeper multi-level financial reconciliation process.

2.9 A clean summary of the two money flows is worth preserving

The later business-logic report restates the core business model in a compact form that is consistent with earlier knowledge and worth retaining as a high-level summary:

  • Customer side / money in: FortnoxOrder → FortnoxInvoice
  • Supplier side / money out: FortnoxSupplierInvoice
  • No direct Fortnox-native bridge exists between what the client was charged and what the supplier/platform charged Adgrabber
  • The bridge exists only inside Airtable via sys.orderallocationmapping and sys.fakturamapping, both meeting at Media / Prod New

This does not replace earlier architecture material, but it is a useful concise business-logic summary.

4. Explicitly Excluded from This Delta

The following topics are intentionally omitted from this note because the three later files disagree with each other, present them as inferred, or do not prove them strongly enough to be safe KB facts.

Excluded: exact Airtable view usage by n8n

Some later files describe per-table sync views as the operational drivers. Another later file treats at least some of those views as human-monitoring views or unresolved implementation details. Because of that inconsistency, this delta note does not preserve exact claims about which views n8n truly uses.

Excluded: exact polling-window model

The later files contain useful ideas about time windows, but the interpretation of those windows depends on the disputed view-usage story. Until the n8n workflows themselves are inspected, the exact polling-window model should remain outside the contradiction-free KB layer.

Excluded: confirmed cron/trigger mechanics

One later report infers a cron-style schedule and a likely trigger approach, but the mechanism is not confirmed directly in workflow code. This note therefore avoids presenting trigger logic as settled fact.

Excluded: live status of auto-created orders from brief.ordernr

The later reports suggest a schema path involving Brief. OrderNumber and automated order-shell creation, but they do not prove that workflow is active in current live operations. It is left out here.
