2. Architectural Role and Dependencies

This workflow sits in the middle of the order sync graph. Its role is wider than just "sync one order header." It also controls row reconciliation and article linkage for each row.

DirectionWorkflowPayloadPurpose
Parent -> this component Downloader / WebSocket Handler / Invoice Component { id: <Fortnox order number> }
or { id: <order number>, customer_id: <Airtable customer rec id> }		 Tells the component which single order to fetch and, optionally, gives it an already-resolved Airtable customer id.
This component -> Credentials Workflow 109 No business payload Loads shared config such as Airtable table ids, component workflow ids, and the Fortnox ratelimit proxy id.
This component -> Customer Component Workflow 111 { id: Order.CustomerNumber } Runs only when the parent did not already provide customer_id. Ensures the related Fortnox customer exists in Airtable first.
This component -> Article Component Workflow 110 { id: OrderRow.ArticleNumber } Ensures the row's Fortnox article exists in Airtable so the order row can link to it.
Role relative to the whole model: the parent FortnoxOrder table stores the raw Fortnox order and its serialized OrderRows blob, while the child Fortnox OrderRow table materializes each individual order row as its own Airtable record. This component is the bridge that keeps those two layers in sync.

3. Observed Execution Path

The workflow path visible in the JSON is more complex than the invoice component because it performs row replacement by soft-delete + reactivation rather than only appending fresh child rows.

Parent sends one order id
  -> Credentials (workflow 109)
  -> GET /3/orders/{id} via ratelimit proxy
  -> resolve customer_id
       - if parent already gave one, reuse it
       - else call Customer Component with CustomerNumber
  -> find Airtable FortnoxOrder by DocumentNumber
  -> update or create FortnoxOrder header
  -> compute should_sync flag on returned Airtable order record
  -> if should_sync is false: stop and return order id
  -> find all existing Airtable Fortnox OrderRow records for this order
  -> mark all existing rows Deleted = true
  -> inspect Fortnox OrderRows array
       - if empty: stop and return order id
       - else split rows one by one
  -> for each row:
       - resolve article if ArticleNumber is valid
       - find row by RowId
       - set LineItem Source = Fortnox, Deleted = false, link order and article
       - update existing row or create new row
  -> after last row: return Airtable order id
1
Load shared config and fetch the Fortnox order
The workflow calls the shared credentials workflow, then issues GET /3/orders/{id} through the shared ratelimit proxy.
2
Resolve the customer link
If Start.customer_id already exists, the workflow skips the Customer Component and reuses that Airtable record id. Otherwise it calls the Customer Component with Order.CustomerNumber.
3
Upsert the order header
It finds the Airtable order by DocumentNumber, then either updates the existing record or creates a new one.
4
Gate the row branch with should_sync
After the header write, the workflow builds a small object containing the Airtable order id and a boolean called should_sync derived from fields["Internal Status"] !== "Do not sync".
5
Mark prior child rows as deleted
If row sync is allowed, the workflow finds all existing Airtable order rows for this order using the formula {Order Number Text} = '{{ Start.id }}' and sets Deleted = true on all of them first.
6
Process the current Fortnox row array
If OrderRows.length > 0, it iterates one row at a time. If the array is empty, it exits after the soft-delete pass.
7
Resolve article per row and upsert by RowId
Each row optionally calls the Article Component, then the workflow finds an Airtable row by RowId and either updates it or creates it.
8
Reactivate surviving rows
The row code sets Deleted = false on every row that still exists in the latest Fortnox payload. Rows not recreated remain soft-deleted.

4. Header Record Write Behavior

The header-side part of the workflow is straightforward but still important because it normalizes Fortnox data for Airtable consumption.

Observed elementWhat it doesWhy it matters
Find node Looks up an existing Airtable order by DocumentNumber. That makes DocumentNumber the business identity for the header record.
Code node Converts Fortnox booleans like Cancelled, CopyRemarks, VATIncluded, and Sent into strings and serializes complex objects like EmailInformation, Labels, and OrderRows to JSON strings. This matches the Airtable schema design where several Fortnox structures are stored as text snapshots rather than native nested objects.
FortnoxCustomer assignment The header payload is enriched with FortnoxCustomer = [customer record id]. That ties the order to the already-synced customer master record.
Update / Create Writes the parent record into the Airtable FortnoxOrder table. The order header becomes the anchor for downstream invoice linkage and row rollups.

Important design fact

The workflow does not use @url directly as the upsert key in the n8n component itself. The uploaded component finds records by DocumentNumber. That differs from some broader table-level audit language that describes the general sync model at table level.

Likely reason for the design

Using DocumentNumber inside the component is operationally sensible because parent workflows also identify orders by Fortnox order number, and downstream entities such as invoices link back through that number. The current source set does not explicitly explain why the component chose this key, but it is consistent with the wider order model.

5. Row Reconciliation and Soft-Delete Pattern

This is the most important behavior in the workflow. The component is not simply "append new rows." It first assumes every previously known row may be obsolete, marks them all deleted, then revives only the rows that still exist in the latest Fortnox payload.

StepObserved node(s)Meaning
1Find All RowsReads all Airtable child rows matching {Order Number Text} = Start.id.
2Set Deleted on All Rows + Update All Rows as DeletedSoft-deletes every currently linked row before processing the fresh Fortnox payload.
3JSON Rows + Split RowsExplodes the current Fortnox OrderRows array into one execution item per row.
4Find RowLooks up a row by RowId, the Fortnox row identifier.
5Code RowSets LineItem Source = "Fortnox", clears Deleted, and links the row to both the parent order and optional article.
6Update Row / Create RowReactivates an existing row or creates a brand-new one if Fortnox introduced a new row id.
Confirmed reconciliation model: deletion is handled as a soft-delete, not a hard delete. Missing Fortnox rows remain in Airtable with Deleted = true; surviving rows are explicitly reset to false.

Why this aligns with the field audits

The uploaded OrderRow audit already describes the child table as a hybrid operational table with explicit soft-deletion logic, RowId as the Fortnox row key, and LineItem Source = "Fortnox" written by n8n. The workflow JSON matches that model exactly.

Practical effect of the pattern

This design preserves historical row records, allocations, and operational notes even when Fortnox removes or changes a row. Instead of erasing the Airtable record, the component leaves a visible trace that the row no longer exists in the current Fortnox payload.

7. Confirmed Quirks and Unresolved Questions

7.1 The export is inactive, but the component is still referenced elsewhere

The uploaded JSON proves "active": false. Separately, the broader source set shows workflow 114 referenced as component_order by the Downloader, WebSocket Handler, and Invoice Component. That means the component remains part of the documented dependency graph even though this export was disabled when captured.

7.2 Customer resolution is optimized for chained calls

The workflow explicitly accepts an incoming customer_id from a parent. That is why the Invoice Component can call Order Component without forcing a second customer sync when it already resolved the customer one step earlier.

7.3 The row branch uses a suspicious Internal Status gate

The Order set node computes should_sync = fields["Internal Status"] !== "Do not sync" on the returned Airtable order record. In the currently uploaded schema notes, Internal Status is clearly documented for FortnoxSupplier, not for FortnoxOrder. The current evidence set still does not clearly document a same-named field on the order table. This is a separate question from Order Status — they are not the same field.

Still unresolved: the available sources do not prove whether FortnoxOrder really has an undocumented Internal Status field, whether the field existed in an older schema, or whether this condition now effectively evaluates to true for all orders because the field is absent. This ambiguity remains specifically about Internal Status, not about Order Status.

7.4 Order Status exists in schema but is dormant in production

User verification and the broader schema notes agree on the same point: the FortnoxOrder table does have an Order Status field, but it is empty on production records. That fits the documented interpretation that Order Status belongs to a designed Airtable -> Fortnox push lifecycle (Airtable Draft -> Copy to Fortnox -> Fortnox Order) which is not active in real usage.

For this component, that matters in a very concrete way: the workflow does not read or write Order Status anywhere in the uploaded export. It works by fetching one Fortnox order, upserting the header by Fortnox identifiers, and reconciling child rows. So a blank Order Status is not a sync failure inside this workflow — it is the expected state of an unused lifecycle field.

7.5 The component preserves deleted rows instead of removing them

This is not speculative. The workflow marks all prior rows Deleted = true before rebuilding the live set, and it only flips current rows back to false. That creates a durable audit trail of removed rows in Airtable.

7.6 Why the soft-delete design is probably intentional

Because order rows participate in allocation tracking and operational QA, hard deletion would break downstream visibility. A soft-delete pattern lets the team see that a row disappeared from Fortnox while keeping the historical Airtable record alive. The workflow JSON proves the pattern; the business rationale is an inference from the surrounding schema.

7.7 Push-path fields exist in schema, but current live usage appears pull-dominant

The broader Airtable documentation says the order schema contains push-oriented fields such as Sync Fortnox and lifecycle statuses, while live-data notes say all current order records appear to have come from Fortnox and all current order-row records show LineItem Source = "Fortnox". This component fits the pull/mirror side of that reality, and the blank production Order Status values reinforce that conclusion.

8. Plain-English Verdict

This workflow is the order mirror-and-reconciliation child flow for Fortnox orders.

  • It receives one Fortnox order id from a parent workflow.
  • It fetches the current Fortnox order payload and upserts the Airtable FortnoxOrder header.
  • It then reconciles the child Fortnox OrderRow table by soft-deleting old rows and reactivating or creating the rows that still exist in Fortnox.
  • It optionally ensures the customer and referenced articles exist in Airtable through child component calls.
Bottom line: the uploaded export proves that workflow 114 is designed as a Fortnox -> Airtable order component with explicit row-level reconciliation logic. The production-side observation that Order Status is always blank now fits cleanly into the analysis: it behaves like a dormant push-lifecycle field, not like an active control field for this component. The main unresolved issue that remains is the meaning of the separate Internal Status gate on the order record, because that field is still not clearly documented on FortnoxOrder in the current source set.

End of workflow note.
