Why the Settings table matters

The Downloader is not triggered by Fortnox change events directly. It is driven by a small state record in Airtable containing:

  • boolean flags such as Download Invoices, Download Orders, Download Customers, etc.
  • persisted page counters such as Invoices Downloader Page, Orders Downloader Page, and Supplier Invoices Downloader Page

3. High-Level Workflow Behavior

The Downloader is primarily a batch orchestrator/paginator, not the place where the detailed sync logic lives. It lists Fortnox records page by page, normalizes each listed item's Fortnox ID into a small payload, and delegates the full detail fetch + Airtable upsert to child component workflows.

Trigger from Settings table
  -> Credentials (workflow 109)
  -> Get Settings
  -> decide which downloader branch should run
  -> call Fortnox list endpoint for one entity family
  -> extract entity array from response
  -> split items one by one
  -> pass normalized ID into child component workflow
  -> decide whether the current Fortnox page is final
  -> write next page or next stage back to Settings
Architectural distinction: the child components still perform the detailed record sync. Example: the invoice branch lists invoice summaries via GET /3/invoices, then sends each invoice DocumentNumber to the invoice component, which handles the detailed fetch/upsert.

5. Confirmed Branch Activation Conditions

The workflow JSON makes an important distinction between two families of downloader branches.

5.1 Persisted resume branches

Invoices, Orders, and Supplier Invoices each use an IF node with combineOperation = "any".

  • Invoice branch runs when Invoices Downloader Page > 1 or Download Invoices = true.
  • Order branch runs when Orders Downloader Page > 1 or Download Orders = true.
  • Supplier-invoice branch runs when Supplier Invoices Downloader Page > 1 or Download Supplier Invoices = true.
Corrected interpretation: a page counter greater than 1 is itself a valid start condition for these three branches. So a live value like Invoices Downloader Page = 34 is consistent with the workflow and should be read as a persisted mid-pagination state, not as evidence that the reset logic is missing.

5.2 Non-persisted branches

Customers, Articles, and Suppliers do not have stored Airtable page counters in this workflow. They are started by boolean flags and then loop internally using a temporary CurrentPage variable.

6. Two Different Pagination Patterns

The Downloader does not paginate every entity family the same way.

A. Persisted page-state branches

Invoices, Orders, and Supplier Invoices use page counters stored in the Settings table.

  • The branch reads page N from the Settings record.
  • It processes one Fortnox page per execution path through that branch.
  • If the page is not final, it writes page N + 1 back to Settings.
  • If the page is final, it writes page 1 and hands control to the next entity family.
Settings page counter = N
  -> HTTP list page N
  -> process all items in page N
  -> if not final page: write N+1 back to Settings
  -> if final page: write 1 and enable next branch

B. In-memory looping branches

Customers, Articles, and Suppliers use a temporary field called CurrentPage inside the current execution.

  • The branch begins with CurrentPage = 1.
  • If more pages remain, the workflow loops directly back to the same HTTP node in the same execution.
  • No persistent Airtable page checkpoint is written for these branches.
CurrentPage = 1
  -> HTTP list page 1
  -> process all items in page 1
  -> if not final page: CurrentPage = 2 and loop immediately
  -> continue until final page
  -> then update Settings once to hand off to next branch
Operational consequence: invoice/order/supplier-invoice backfills appear resumable across separate runs because their page number is persisted, while customer/article/supplier backfills appear to rely on finishing that entity family within one execution chain.

7. Corrected Explanation of What the Stored Page Fields Mean

The persisted page fields should be read as the next page to fetch / the current resume checkpoint, not as “the total number of Fortnox pages.”

FieldWhat it meansWhat it does not mean
Invoices Downloader PageThe invoice page the workflow should fetch next if the invoice branch runsNot the total number of invoice pages that exist in Fortnox
Orders Downloader PageThe order page the workflow should fetch next if the order branch runsNot the total number of order pages that exist
Supplier Invoices Downloader PageThe supplier-invoice page the workflow should fetch next if that branch runsNot the total number of supplier-invoice pages that exist

Important correction about reset behavior

These fields do not always sit at 1. They are reset to 1 only when the workflow reaches the branch’s explicit final-page reset node. Until then, they can legitimately remain at 2, 12, 34, etc.

Confirmed from JSON:
  • Update Invoices CurrentPage writes Invoices Downloader Page = @CurrentPage + 1.
  • Reset Invoices CurrentPage writes Invoices Downloader Page = 1 and enables orders.
  • The same structural pattern exists for orders and supplier invoices.

8. User-Reported Live State and Revised Interpretation

8.1 User-reported live observation

During this discussion, the user reported that the current Airtable Settings record shows approximately:

  • Invoices Downloader Page = 34 (example live value)
  • Orders Downloader Page = 1
  • Supplier Invoices Downloader Page = 1

This is a runtime observation from the live base, not something directly present inside the uploaded JSON export.

8.2 Revised interpretation of those live values

Given the confirmed IF-gate logic, the most likely interpretation is:

  • Invoices Downloader Page = 34 means the invoice branch is currently in a persisted mid-pagination / resume state.
  • Orders Downloader Page = 1 means the orders branch is not currently parked mid-pagination.
  • Supplier Invoices Downloader Page = 1 means the supplier-invoice branch is not currently parked mid-pagination.

This inference is strongly supported by the workflow structure because the invoice branch can start from page > 1 even if the boolean flag is false. However, the uploaded files do not prove why the live invoice branch is parked at page 34 at that exact moment. The JSON shows that this state is possible; it does not contain a runtime execution log explaining when or why it happened.

What is proven vs. not proven here:
  • Proven: invoice/order/supplier-invoice branches can resume when their page counter is greater than 1.
  • Proven: they only reset to 1 on the explicit final-page reset path.
  • Not proven from uploaded files: whether the live invoice branch is paused because of an interruption, because the cycle is intentionally mid-run, or because the next trigger has not yet fired.

11. Inferences That Remain Plausible but Not Fully Proven

11.1 The Downloader likely advances itself by retriggering from Settings updates

The workflow both watches the Settings table and writes back to the same Settings record. That strongly suggests a chained execution model:

  1. one run finishes a page or a stage,
  2. Update Settings writes the next flag/page state,
  3. the Airtable trigger sees the modified settings record on its next poll,
  4. the workflow starts again and continues from the new state.

This is highly plausible from the structure, but the uploaded files do not contain execution logs proving that timing sequence.

11.2 The workflow is designed as a controlled sweep / reconciliation cycle

Unlike the WebSocket Handler, which reacts to Fortnox events in real time, the Downloader walks Fortnox list endpoints in a fixed entity order. That makes it suitable for backfill, reconciliation, or catch-up processing.

11.3 Supplier-invoice file syncing is intentionally suppressed in the bulk path

The supplier-invoice branch sends sync_files = "no" into the child workflow. Compared with the WebSocket path discussed separately, this strongly suggests that invoice-file attachment syncing is intentionally skipped during bulk sweeps.

11.4 The disabled Schedule Trigger is probably legacy or fallback

Because the schedule trigger exists but is disabled, it likely reflects an older design or a fallback start mechanism that is not currently used in production.

12. Current State of Understanding

  • The Downloader is an active batch orchestrator workflow with ID 118.
  • Its job is to page through Fortnox list endpoints and delegate each listed record into entity-specific child components.
  • The workflow is controlled by an Airtable Settings record that acts like a small state machine.
  • The branch order is confirmed as Invoices -> Orders -> Customers -> Articles -> Supplier Invoices -> Suppliers -> repeat.
  • Invoices, Orders, and Supplier Invoices use persisted page counters; Customers, Articles, and Suppliers use in-memory page loops.
  • A stored page value greater than 1 is a real resume condition for the persisted branches.
  • The missing workflow 109 is very likely the shared config/catalog workflow named Credentials.
  • The user’s live observation that invoices can sit at page 34 is consistent with the workflow and has now been incorporated into the analysis as a user-reported runtime state plus explicitly labeled interpretation.
