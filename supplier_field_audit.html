TABLE FortnoxSupplier — Full Field Audit

Table ID: tblSFdN7U9Bri8zwr  |  Base: DEV. Betaportal Adgrabber 2.0 (app01yDjf98U6ejKs)  |  Total fields: 69  |  Audit date: 2026-03-25

This table stores supplier master data. Records come from two sources: (1) suppliers synced automatically from Fortnox via n8n, and (2) platform suppliers created manually (media platforms such as LinkedIn, Facebook, Google — not registered in Fortnox). Both types share one table but differ significantly in which fields are populated.

Sync gate for invoices: The n8n invoice sync reads each supplier's Internal Status field. Only suppliers set to "Sync" have their invoices pulled from Fortnox. Suppliers with "Do not sync" or no status set are skipped entirely.

A — Fortnox-Side Fields  Origin: Fortnox / n8n sync

These fields all originate from the Fortnox accounting system. n8n polls GET /3/suppliers and upserts records in Airtable using FIELD @url as the upsert key.

⚠️ Any manual edit to an A-field in Airtable will be silently overwritten on the next sync. Changes to supplier master data must be made in Fortnox first.

Unlike FortnoxSupplierInvoice (where Fortnox auto-calculates totals, balances, and voucher assignments), the FortnoxSupplier fields are almost entirely master data entered by staff in Fortnox. There are no Fortnox-computed financial calculations here. The A1/A2 split reflects only whether the value is assigned automatically by Fortnox vs. explicitly entered by staff.

Note on sparse fields: Many A2 fields (Address2, CostCenter, Email, Fax, OurCustomerNumber, PG, Phone2, Project, VATType, VisitingAddress/City/Country/ZipCode, WWW, YourReference) are rarely or never populated. Fortnox's supplier record supports them, but Adgrabber staff simply do not fill every optional field. Their presence in Airtable reflects the full API spec, not actual usage.

A1 — Auto-generated identifiers  Fortnox / n8n assigns · not user-chosen

These two fields are assigned automatically — SupplierNumber by Fortnox (auto-incremented when a new supplier is created without specifying one), and @url constructed by n8n from the SupplierNumber as the canonical API endpoint. Staff do not choose these values; they come into existence when the supplier record is created in Fortnox.

A2 — Staff-entered master data in Fortnox  Staff entered this in Fortnox · n8n syncs it here

These fields are the supplier's master data record in Fortnox — contact info, banking details, accounting defaults, etc. Staff at Adgrabber (or the supplier's own contact at Adgrabber) maintain this data in Fortnox. n8n copies whatever is stored there into Airtable on each sync cycle. If a value looks wrong in Airtable, the correction must be made in Fortnox.

These fields are set directly in Airtable by Adgrabber staff. They control sync behavior, mark special supplier types, and capture operational details that do not exist in Fortnox. They are not touched by n8n and will not be overwritten by sync.
