Scope of this note. This document summarizes the analysis of the active workflow Adgrabber Fortnox WebSocket Handler v3.0.0, the role of the custom WebSocket trigger, the branch structure, and the current understanding of the missing helper workflow 109 referenced as Credentials.

2. Entry Trigger

The first node is a custom community node of type n8n-nodes-rhgroup.webSocketTrigger. In a fresh local n8n instance it may appear as a question mark until the package is installed.

PropertyValue
Trigger endpointwss://ws.fortnox.se/topics-v1
Auth sourceFortnox OAuth credential selected on the WebSocket Trigger node
Trigger roleOpen socket, authenticate tenant, register topics, subscribe, then emit incoming Fortnox events to the workflow.

Handshake code meaning

  1. add-tenants-v1 sends the OAuth access token and client secret to Fortnox for tenant/session registration.
  2. add-topics-v1 registers which event domains the workflow wants to receive.
  3. subscribe-v1 starts the live event stream.
  4. If any Fortnox response does not return result === "ok", the node throws an error and the trigger does not start.
Subscribed Fortnox topics:
- articles
- invoices
- orders
- customers
- suppliers
- supplier-invoices

3. High-Level Workflow Behavior

The WebSocket Handler is primarily an orchestrator/router, not the place where the full data-sync logic lives. It receives a Fortnox event, normalizes the relevant ID, and then delegates the actual record fetch/upsert work to downstream component workflows.

Fortnox event
  -> WebSocket Trigger
  -> Credentials (workflow 109)
  -> topic/type routing
  -> child component workflow
  -> Airtable sync inside child workflow

5. What Workflow 109 Appears To Be

Important: workflow 109 itself was not present in the uploaded workflow JSON set inspected in this discussion.

The node named Credentials calls workflow 109. Based on how other workflows read its output, the missing workflow does not appear to be a business-action workflow. It is most likely a config/helper workflow that returns a JSON object containing shared IDs and references.

For the WebSocket Handler specifically, 109 is expected to return:

{
  "component_invoice": "...",
  "component_order": "...",
  "component_customer": "...",
  "component_supplier": "...",
  "component_supplier_invoice": "...",
  "component_article": "..."
}

7. Practical Reconstruction Strategy

Most likely feasible: recreate workflow 109 manually as a simple helper workflow that returns a single JSON object with the required config values.

If all workflows are imported into the same local n8n database, a replacement 109 can be created with:

  • the new local workflow IDs for each child component workflow
  • the relevant Airtable base/table IDs
  • other shared constants such as ratelimit_id, if known

This would not recreate stored n8n credentials/secrets by itself, but it would likely restore the routing/config layer expected by the imported workflows.

8. Current State of Understanding

  • The WebSocket Handler is an active parent/orchestrator workflow.
  • Its job is to receive Fortnox real-time events and route them into child sync components.
  • The missing workflow 109 is very likely a shared config/catalog workflow named Credentials.
  • Within the WebSocket Handler specifically, 109 appears to be used only to provide child workflow references.
  • Across the wider workflow set, 109 is also expected to return Airtable IDs and other shared config values.

End of note.
