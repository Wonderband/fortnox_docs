Allocation Mapping — Revenue & Cost Attribution

sys.orderallocationmapping  |  sys.fakturamapping  |  base app01yDjf98U6ejKs

The core problem these tables solve: Fortnox records revenue when an invoice is issued and records cost when a supplier invoice arrives. But Adgrabber sells ad campaigns that run across multiple months. A single order can span October–December; a single supplier invoice can cover costs for several campaigns. Airtable needs a way to split both revenue and cost into the right month and the right campaign. TABLE sys.orderallocationmapping does this for revenue; TABLE sys.fakturamapping does this for cost.

Connection Map — How The Tables Relate

TABLE Media / Prod New is the central join point. It represents a single campaign line item — e.g. "Google Ads for Atteviks Bil, Oct–Dec 2023". Both mapping tables point to it, allowing Airtable to compute margin per campaign line: Revenue (from orderallocationmapping) minus Cost (from fakturamapping).

TABLE sys.orderallocationmapping — Monthly Revenue Split

What it does

A Fortnox order row has a single total (e.g. 9,600 SEK for a 3-month Google Ads campaign). That total needs to be split into monthly revenue — Oct: 3,200 SEK, Nov: 3,200 SEK, Dec: 3,200 SEK. TABLE sys.orderallocationmapping holds one record per month-slice. None of this data goes back to Fortnox — it is purely internal Airtable bookkeeping for campaign reporting.

Why two allocation records per OrderRow? Each OrderRow total is split into two allocation types:

1. "Pay for mediaköp" — the actual media spend Adgrabber pays out to the platform (Google, Facebook, etc.) on the customer's behalf. In the example: 2,560 SEK (Oct), 2,458 SEK (Nov/Dec).
2. "Pure profit – no spend" — the agency margin on top, with no outgoing cost. In the example: 640 SEK (Oct), 614 SEK (Nov/Dec).

The two amounts always sum to the OrderRow FIELD Total excl. VAT. This split lets Adgrabber track gross margin per campaign line: Pure profit records represent what the agency keeps; Pay for mediaköp records represent what flows out to the platform.
Staff creates these records manually. When a campaign is set up, staff open the TABLE sys.orderallocationmapping interface and create allocation records linking each month's OrderRow to the campaign line. No automation writes to this table.

TABLE sys.fakturamapping — Cost Attribution Per Campaign

What it does

A supplier invoice from Google or Facebook covers ad spend for multiple campaigns. TABLE sys.fakturamapping holds one record per campaign that was served by that invoice. It answers: "Of this 15,000 SEK Google invoice, how much belongs to campaign M8347 and how much to M8350?"

A second use: monthly cost split. If one invoice covers October–November spend, staff can create two fakturamapping records for the same invoice — one per month — to track cost by period. However the primary split is per campaign.

Key Fields

Fortnox SupplierinvoiceLinked record → TABLE FortnoxSupplierInvoiceWhich supplier invoice this cost slice comes from. One invoice can have many fakturamapping records.
Mediakop/ProductionLinked record → TABLE Media / Prod NewWhich campaign line item bears this cost.
KampanjLinked record → TABLE KampanjerParent campaign.
Belopp ex momsCurrencyThe SEK cost (excl. VAT) attributed to this campaign from this invoice.
Platform Import FileLinked record → TABLE Platform Import FilesOptional link to the PDF billing file that was the source for this cost attribution.

Why three invoices and not one? Google bills monthly. Each month's ad spend produces a separate supplier invoice in Fortnox. Staff imports the monthly Google billing PDF into TABLE Platform Import Files, then creates one fakturamapping record per invoice, per campaign that ran during that month. If five campaigns ran on Google in October, the October Google invoice gets five fakturamapping records — one per campaign, amounts summing to the invoice total.

Typical Staff Workflow — From Campaign Setup to Monthly Close

1
Campaign setup. Staff creates a TABLE Kampanjer record and adds TABLE Media / Prod New line items — one per media channel. For each line item, staff manually sets FIELD Brief. Kundens mediabudget (the customer order amount, e.g. 9,600 SEK for a 3-month Google Ads line). FIELD Total Ek. budget. Intäkt and FIELD Budget are computed formulas — they resolve automatically from that manual input. Neither sys table writes to any budget field; all budget values originate here, at campaign setup.
2
Order allocation. Staff links a Fortnox order row to the campaign in TABLE sys.orderallocationmapping, creating one record per month that the campaign runs and splitting the order row total across those months. Each record links to the order row AND the Media / Prod New line item.
⚠ Schema note: The schema includes a push path where staff creates order rows directly in Airtable with FIELD LineItem Source = "Airtable users". Confirmed from live data: every Fortnox OrderRow record has LineItem Source = "Fortnox" — the push path has never been used. This is one of three live-data proofs that the Airtable → Fortnox push workflow is inactive (see Fortnox ↔ Airtable — Helicopter View for the full list of proofs).
3
Monthly close — supplier invoices arrive. At end of each month, Google, Facebook, etc. issue billing invoices. These arrive in Fortnox (as supplier invoices). n8n syncs them into TABLE FortnoxSupplierInvoice.
4
Import & reconcile. Staff downloads the platform billing PDF and uploads it to TABLE Platform Import Files, links it to the TABLE FortnoxSupplierInvoice record. Then for each campaign that ran that month on that platform, staff creates a TABLE sys.fakturamapping record: link to the invoice, link to the Media / Prod New line item, enter the SEK amount for that campaign. The sum of all fakturamapping amounts for one invoice must equal the invoice total.
5
Reporting. TABLE Media / Prod New FIELD Margin and FIELD Budget Overrun are now live. Management can see, per campaign line, the actual margin and whether costs exceeded plan. Nothing here goes back to Fortnox — Fortnox only sees the invoices themselves, not the internal campaign attribution.
Neither table syncs with Fortnox. TABLE sys.orderallocationmapping and TABLE sys.fakturamapping are purely internal Airtable records. They exist because Fortnox is a financial accounting tool — it records totals per invoice. It has no concept of "which part of this invoice belongs to campaign M8347 in October." That attribution layer lives entirely in Airtable and is maintained by Adgrabber staff.
