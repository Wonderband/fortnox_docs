2. Architectural Role and Dependencies

This workflow sits in the cost-side Fortnox branch. It does not decide campaign attribution and it does not create mapping rows itself. Its narrower job is to keep one supplier invoice header mirror available in Airtable, with the correct supplier link and an optional file-sync follow-up.

Whether the file-helper branch runs or not, the workflow's success contract is minimal: it returns only the Airtable supplier-invoice record id.

4. Header Record Write Behavior

The write-side logic is more revealing than it first appears. The important design choices are which identity key it uses, what it normalizes, and what it intentionally leaves as a JSON blob.

Observed elementWhat it doesWhy it matters
Find Looks up an existing Airtable supplier invoice by GivenNumber. That makes GivenNumber the business identity used inside this component.
Code node Converts Balance to a string, parses Total and VAT into numeric values, stringifies SupplierInvoiceRows and Vouchers, and sets FortnoxSupplier to the Airtable supplier record id returned by workflow 112. This is the bridge between the Fortnox response shape and the Airtable schema described in the field audit.
Exists IF node Tests whether the list result contains an existing Airtable record. This is the upsert gate between update and create behavior.
Update / Create Writes the parent record into the Airtable FortnoxSupplierInvoice table. The invoice becomes available as the cost-side header record that downstream mapping and accrual logic can reference.
JSON Supplier Invoice Passes the written Airtable record forward unchanged. The next gate uses Airtable-level fields from the written record rather than the raw Fortnox response.

Important design fact

The component does not explode SupplierInvoiceRows into child records here. It stores the row payload as a JSON string on the header record and leaves downstream allocation or file logic to other workflows.

Likely reason for the design

This keeps the component narrowly focused on header-level mirroring. Since the table's real business use is mapping the invoice total into sys.fakturamapping and later into Media / Prod New, a simple header sync is enough to preserve the accounting document while more opinionated parsing can remain separate.

5. Optional Invoice-File Branch

The most distinctive behavior after the header upsert is the optional handoff into the invoice-file helper flow. This is not mandatory for every run.

1Should Download InvoiceChecks two conditions: Supplier Internal Status String.trim() != "Do not sync" and Start.sync_files != "no".

3Supplier Invoice File Component OutputDelegates the attachment/PDF work to the dedicated file-helper flow rather than handling file retrieval in this component.
4Component Output / Component Output1Both success endings return the Airtable supplier-invoice id; the only difference is whether the file-helper side effect was attempted first.
Confirmed branching model: header sync and file sync are deliberately separated. The invoice header can still be created or updated even when file download is skipped.

What the file gate definitely means

The workflow does not assume every supplier invoice must immediately download a file. The caller can explicitly suppress file sync with sync_files = "no", and the workflow also checks the supplier-status-derived Airtable field before it attempts the file branch.

Practical effect of the design

This makes the component reusable in two modes: a fast header-only sync mode and a fuller mode that also tries to attach invoice files or PDF references. That is useful when a parent workflow wants to reduce runtime or avoid repeated file-fetch work.

7. Confirmed Quirks and Unresolved Questions

7.1 The export is inactive, but the component is still part of the documented dependency graph

The uploaded JSON proves "active": false. Separately, the broader source set still treats supplier-invoice sync as an active architectural concern and documents this component's role in the Fortnox-to-Airtable invoice path. So the export was captured in an inactive state, but the workflow remains architecturally meaningful.

7.2 The gate for blank supplier status is looser than the field-audit wording

The broader docs describe the supplier-invoice flow as running only when supplier status is Sync. But the actual exported logic is looser: the upstream supplier component returns should_sync unless the status is explicitly "Do not sync", and the file-download branch repeats that same relaxed check on Supplier Internal Status String. That means blank status appears to pass the current logic unless another external control exists.

7.3 The component uses GivenNumber as its invoice identity key

The workflow finds existing records by GivenNumber, not by a Fortnox URL field. That is consistent with the table docs, which describe GivenNumber as the Fortnox document number used as the practical business identity.

7.4 There is no visible Airtable-only fallback path if the Fortnox fetch fails

Unlike the Customer Component, this workflow has no separate error-output branch and no alternative Airtable lookup path after the HTTP request. From the exported graph, a Fortnox fetch failure would stop the normal path rather than degrade gracefully into a lookup-only recovery.

7.5 Blocked suppliers still reach the supplier component before the invoice stops

The Should Sync gate sits after the Fortnox invoice fetch and after the Supplier Component call. So even when an invoice is ultimately blocked from being written, the workflow has already fetched the Fortnox invoice payload and already refreshed or resolved the supplier master record.

7.6 Why the design likely looks this way

The workflow appears optimized for reliable supplier linkage and minimal parent contracts rather than for strict policy isolation. By resolving the supplier first, it guarantees the supplier record is fresh and returns a simple allow/deny signal to the invoice path. The tradeoff is that the gate does not prevent all upstream work; it only prevents the invoice header write.

8. Plain-English Verdict

This workflow is the Fortnox supplier-invoice header sync component with a supplier-policy gate and an optional file-helper handoff.

Its core job is simple: fetch one supplier invoice from Fortnox, make sure the supplier exists in Airtable, stop if that supplier is blocked from sync, otherwise create or update the FortnoxSupplierInvoice header record and optionally continue into the invoice-file workflow.

It is not the cost-attribution engine. It does not create sys.fakturamapping rows, does not allocate costs into Media / Prod New, does not process platform-import invoices, and does not run accrual logic. It is the document-mirroring step that keeps the cost-side source record available so those later workflows can function.

Bottom line: if you want to understand where one Fortnox supplier invoice becomes one Airtable supplier-invoice record, this is the component. If you want to understand how that invoice gets mapped to campaigns, accruals, or PDF/file attachments, you need the downstream mapping and file-helper flows in addition to this one.
